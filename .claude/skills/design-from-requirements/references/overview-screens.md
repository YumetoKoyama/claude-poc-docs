## screens/共通レイアウト.md

> 全画面に共通するアプリシェル（ヘッダー／フッター／グローバルナビ／サイドバー等）を 1 枚で設計する。各画面 md は本ファイルを参照し差分のみ記述する。共通シェルに乗らない画面（ログイン等の認証前画面）は「認証前独立レイアウト」として区分を明記する。FE 実装方式（共通コンポーネント・状態管理）は `フロントエンド共通設計.md` に委ねる。

```markdown
# 共通レイアウト設計

## ID 凡例

| ID | 意味 |
|----|------|
| SCR-XXX | 画面 ID |
| (operationId) | API 操作 ID（`api/*.yaml`） |

## レイアウト区分

| 区分 | 説明 | 適用画面 |
|------|------|---------|
| 共通シェル | ヘッダー＋（サイドバー）＋メインの共通枠 | SCR-002, SCR-003, ... |
| 認証前独立レイアウト | 共通シェルに乗らない（ログイン・パスワードリセット等） | SCR-001, ... |

## 共通シェル構成

\`\`\`mermaid
flowchart TB
    Header["ヘッダー (AppHeader)"]
    Nav["グローバルナビ / サイドバー"]
    Main["メインコンテンツ (各 SCR)"]
    Footer["フッター"]
    Header --> Main
    Nav --> Main
    Main --> Footer
\`\`\`

## 共通シェルの構成要素

> ヘッダー等に表示する項目は **必ずデータ源（operationId.フィールド）を明示**する（共通シェル項目は本ファイルで一括管理し、画面ごとに重複させない）。ログイン直後の取得経路はログインシーケンス（`sequences/*.md`）と整合させる。

| 要素 | 表示内容 | データ源（operationId.フィールド） | 表示条件（ロール等） | 挙動・遷移 |
|------|---------|--------------------------------|------------------|-----------|
| ヘッダー: アプリ名/ロゴ | 固定 | — | 全ロール | クリックでダッシュボード(SCR-XXX) |
| ヘッダー: ユーザー名 | ログインユーザー名 | `login.user.displayName` / `me.displayName` | 認証済 | — |
| ヘッダー: テナント名 | 所属テナント名 | `me.tenantName` | 認証済 | — |
| ヘッダー: ユーザーメニュー | プロフィール/ログアウト | — | 認証済 | ログアウト → 認証破棄 → SCR-001 |
| グローバルナビ | メニュー項目 | 静的 or `me.roles` で出し分け | ロール依存 | 各 SCR へ遷移 |
| 通知ベル（任意） | 未読件数 | `notifications.unreadCount` | 認証済 | 通知一覧へ |

## 共通の挙動

- 認証切れ（401）時のリダイレクト先（例: SCR-001 ログイン）と再ログイン後の復帰方針。
- 権限により非表示にするナビ項目（要件 `権限マトリクス.md` と整合）。
- グローバルエラー表示（トースト等）の表示位置。実装方式は `フロントエンド共通設計.md` に従う。
```

ファイル名規約: `[SCR-ID]-[画面名].md`（例: `SCR-001-ログイン.md`、`SCR-003-アイテム一覧.md`）。SCR-ID は大文字、画面名は業務用語の日本語で記述する。

- 画面 ID: SCR-001
- URL パス:

- 所属レイアウト: （共通シェル / 認証前独立レイアウト のいずれか。共通シェルの場合は構成・表示項目を `screens/共通レイアウト.md` に委ね、本画面では差分のみ記述する）

## 入力

| 項目名 | 型 | 必須 | 説明 |
|-------|----|------|------|
|       |    |      |      |

## 出力・表示内容

| 項目名 | 型 | データ源（operationId.フィールド） | 説明 |

## API 対応

| operationId | HTTP メソッド | パス | 概要 |

## フロントエンド共通設計.md

> FE の横断的関心事（共通 UI コンポーネント・API クライアント・エラー表示・共通バリデーション・認可ガード・状態管理方針）を設計する。技術スタックの具体（FW・ライブラリ・バージョン）は再掲せず `claude-poc-frontend/.claude/rules/frontend-*.md`（正典）を参照する。界面契約（ErrorResponse・コード値）は `api/_common.yaml`・要件 `コード値定義.md` を正典とし、FE はそれに対応づけるのみで再定義しない。

```markdown
# フロントエンド共通設計

## 共通 UI コンポーネント

| コンポーネント | 役割 | 主な props / 状態 | 使用画面 |
|--------------|------|-----------------|---------|
| Button / Form / Input | 入力共通 | ... | 全画面 |
| Table / Pagination | 一覧表示 | `PageMeta`（`_common.yaml`）に対応 | 一覧系 SCR |
| Modal / Toast | 確認・通知 | ... | 全画面 |

## API クライアント

- 共通クライアントの責務: ベース URL・JWT 付与・共通ヘッダ・タイムアウト・リトライ方針。
- 認証トークンの保持・更新（リフレッシュ）方針は `セキュリティ設計.md` の JWT ライフサイクルと整合させる。
- 401（認証切れ）/ 403（権限不足）/ 404（越境）のハンドリング方針（リダイレクト・トースト・画面内エラー）。

## エラー表示マッピング（界面契約 → 表示）

> 正典は `api/_common.yaml` の `ErrorResponse`（`code` / `details[].reason`）と要件 `メッセージ一覧.md`（MSG-XXX）。FE はコードを表示文言へ対応づけるのみで、コード体系・文言を新規定義しない。

| ErrorResponse.code | 対応 MSG-XXX | 表示方法（トースト/フィールド） |
|--------------------|-------------|------------------------------|
| VALIDATION_ERROR | MSG-XXX | フィールド直下に `details[].reason` を表示 |
| FORBIDDEN | MSG-XXX | トースト |
| NOT_FOUND | MSG-XXX | 画面内エラー |

## 共通バリデーション

- クライアント側バリデーションの範囲（即時フィードバック）と、サーバ側を正とする方針。
- 文字数・必須・enum は API YAML（`required`/`maxLength`/enum）・要件 `コード値定義.md` と一致させる（二重定義しない）。

## ルーティング・認可ガード

- 認証ガード（未認証 → SCR-001）/ ロール別ガード（要件 `権限マトリクス.md` と整合）。
- 共通シェル（`screens/共通レイアウト.md`）に乗せる画面と認証前画面のルート構成。

## 状態管理方針

- グローバル状態（認証ユーザー・テナント・通知）とローカル状態の切り分け。具体ライブラリは `claude-poc-frontend/.claude/rules/frontend-*.md` を参照。
```
