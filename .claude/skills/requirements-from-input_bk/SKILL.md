# 入力から要件定義書を作成する

> **パス解決（マルチリポジトリ対応）**: 本スキル内の `docs/requirements/`・`docs/design/`・`docs/test/` は **docs リポジトリ（claude-poc-docs）ルート相対**のパスを指す。
> - docs リポジトリをカレントとして実行している場合: そのまま使う。
> - 親アンブレラ（claude-poc-rules）から実行している場合（カレント直下に `claude-poc-docs/` が存在する場合）: これらすべてのパスに `claude-poc-docs/` を前置して読み書きする。
> - CI（子リポジトリ単体のチェックアウト）で docs リポジトリが存在しない場合: workflow が追加チェックアウトした docs のパスを使う。それも無い場合は Issue 本文に埋め込まれた設計情報を入力とし、原本の参照が必要なら中断して人間に確認する。

この skill の役割は、ユーザーから渡された非定型な要望・要件素材を、設計工程が安定して読み取れる構造化要件定義書へ整理することです。設計や実装方式の決定には進みません。

要件素材入力: $ARGUMENTS

## 成果物

要件定義書は 1 ファイルにまとめず、以下の構成で `docs/requirements/` 配下に分割して出力する。

- `docs/requirements/概要.md` — 背景・目的・スコープ・対象外・主要ステークホルダ
- `docs/requirements/業務ルール.md` — 業務ルール・制約・例外処理方針
- `docs/requirements/functional/[機能名].md` — 機能要件（**機能単位で 1 ファイル**、業務用語の日本語名。例: `アカウント登録.md`）
- `docs/requirements/ユースケース図.md` — 全アクター × 全ユースケースの俯瞰図（**1 枚必須**、UC-XXX 採番、Mermaid `flowchart`）
- `docs/requirements/activities/[フロー名].md` — 業務アクティビティ図（**業務フローごとに 1 ファイル**、業務用語の日本語名。例: `案件成約フロー.md`、ACT-XXX 採番、Mermaid `flowchart`）
- `docs/requirements/画面一覧.md` — 画面一覧（画面 ID 付き）と画面遷移図（Mermaid）
- `docs/requirements/データモデル.md` — **概念**データモデル（主要エンティティ ENT-XXX、概念 ER 図、状態遷移 ST-XXX）。物理テーブル設計は後続フェーズに委ねる
- `docs/requirements/外部インターフェース一覧.md` — 外部 IF（メール送信・バッチ・外部システム連携、EXT-XXX 採番）。第 1 版で対象外とする IF も明示する
- `docs/requirements/移行要件.md` — 初期データ・マスタデータ・既存データ移行（MIG-XXX 採番）。**移行なしと判断する場合もその旨を明記**
- `docs/requirements/権限マトリクス.md` — ロール × ユースケース/リソースの操作可否一覧 + テナント越境拒否条件（「ロール 1 種・認可制御なし」の場合もその旨を明記）
- `docs/requirements/メッセージ一覧.md` — エラー/警告/情報/確認/バリデーション/通知の利用者向け表示文言（MSG-XXX 採番）。未確定文言は オープン課題.md（Q-MSG*）へ
- `docs/requirements/コード値定義.md` — 業務区分値（ステータス・種別）の概念レベル正典（「区分値なし」の場合もその旨を明記）
- `docs/requirements/通知・文面定義.md` — 通知タイプ・送信メールの件名/本文/差込変数（メッセージ一覧.md に統合しても可。通知・メールが無い場合もその旨を明記）
- `docs/requirements/非機能要件.md` — 非機能要件（性能・可用性・セキュリティ・運用・データ保持・アクセシビリティ。ベースライン項目は TBD を残さず オープン課題.md へ逃がす）
- `docs/requirements/ブランドガイドライン.md` — ブランド / UI ガイドライン（**任意成果物**。既知範囲で記述し、不明部分は オープン課題.md へ）
- `docs/requirements/用語集.md` — 用語集
- `docs/requirements/オープン課題.md` — 未解決事項・確認待ち事項（**設計着手前クローズ必須課題のクローズ手順章を含む**）

## 指示

1. 引数で渡されたパスまたは要約から要件素材を読む。複数素材があれば全て読み合わせる。
2. 各ファイルの構造には [requirements-template.md](requirements-template.md) を使う。
3. 機能要件は機能単位でファイルを分ける。機能が複数ある場合は `docs/requirements/functional/` 以下に機能ごとのファイルを作成する。
4. 画面ごとに **画面 ID（SCR-001 形式・3 桁ゼロ埋め）** を採番する。サブ画面・モーダルは `SCR-001-01` の枝番で表現する。
5. `画面一覧.md` には、画面 ID・画面名・主要遷移元/遷移先を一覧化したうえで、Mermaid 記法による画面遷移図を必ず含める。
5.1. **ユースケースごとに ユースケース ID（UC-001 形式・3 桁ゼロ埋め）を採番** し、`ユースケース図.md` に全アクター × 全ユースケースの俯瞰図（Mermaid `flowchart`、システム境界を `subgraph` で表現）を 1 枚作成する。`概要.md` の主要ユースケース表と `functional/[機能名].md` の関連ユースケース欄からは UC-XXX で引用する。
5.2. 業務プロセスに分岐や複数アクター間の引き継ぎが存在する場合、**業務フローごとに アクティビティ ID（ACT-001 形式・3 桁ゼロ埋め）を採番** し、`activities/[フロー名].md` に Mermaid `flowchart`（判断ノードは `{ ... }`）でフローを描く。フロー名は業務用語の日本語名（例: `案件成約フロー.md`）。ステップごとに関連 UC-XXX / BR-XXX / AC-XXX を明示する。
6. 業務ルールはフラットに並べず「条件 → 結果」「入力 → 検証 → メッセージ」の形に分解する。
7. 用語の揺れは禁止。プロジェクト固有用語は `用語集.md` に登録し、本文では用語表記をそろえる。
8. ファイル名は固定部・可変部とも日本語名で統一する。可変部（`functional/[機能名].md` の `[機能名]` 部分）は業務用語の日本語名とする（例: `アカウント登録.md`, `仕事検索.md`）。
9. 既にファイルが存在する場合は該当ファイルのみ更新し、無関係なファイルは書き換えない。
10. 設計上の実装手段（フレームワーク選定、テーブル設計、API パス確定、Controller/Service 分割など）は書かない。それは後続の `design-from-requirements` の責務である。
11. 暗黙補完は禁止。曖昧・矛盾・未確定は `オープン課題.md` に明示し、本文には書かない。代わりに `> **[要確認]** {確認内容} / 選択肢A: ... / 選択肢B: ... / 影響範囲: ... / 確認期限: 設計着手前 / DB 設計前 / UI 実装前` の形式で本文に残し、同じ内容を `オープン課題.md` にも転記する。
12. ブランド方針・トーン・想定デバイス・ペルソナなど UI 設計に関わる情報が要件素材から得られた場合は `docs/requirements/ブランドガイドライン.md` に集約する。素材に該当情報がない場合はファイルを作らず、その旨を `オープン課題.md` に列挙する（無理に推測しない）。
12.1. **ガードレール（MUST NOT）** — 詳細は [`requirements-guardrails`](../requirements-guardrails/SKILL.md) を参照。要件として断定してはいけないものは以下の通り。該当する場合は上記 11 の `[要確認]` 形式で残し、`オープン課題.md` に転記する。
    - 認証系: パスワードポリシー・ロック条件・セッションタイムアウト・同時ログイン制御を未定義のまま記述しない
    - データ定義系: ログイン ID の一意性スコープ・金額/容積の単位・マスタ初期値を未定義のまま記述しない
    - 業務制約系: 締切境界判定（00:00 は当日扱い／翌日扱い）・競合解消ルール・上書き可否を未定義のまま記述しない
    - 運用系: バッチ実行時刻・通知保持期間・対象ブラウザ/デバイス・バックアップ頻度・障害通知経路・ログ保管期間を未定義のまま記述しない
    - 例外系: 競合制御（同時操作）・取消可否・差戻し要否・削除中データ（論理削除〜物理削除間）の画面表示方針を **正常系のみで閉じない**
    - 設計情報の混入禁止: フレームワーク名・ライブラリ名（例: Spring Security、React）・DB カラム名・API パスを要件として記述しない
    - 分離必須: ブランドガイドライン・UI ライブラリ・多言語・トーン & ボイスを業務要件と同列に混在させない
12.2. **文書間整合（クロスリファレンス）** — 1 つの資料を作成・更新したら、影響を受ける他資料との整合を確認する。詳細手順は [`requirements-guardrails/references/consistency-checklist.md`](../requirements-guardrails/references/consistency-checklist.md) を参照。最低限、以下は毎回確認する:
    - **ステータス遷移**: 用語集のステータス定義 ＝ ステータス遷移図 ＝ 業務ルールの遷移条件 ＝ 画面・通知の操作条件。**「第 1 版では使用しない」と定義したステータスは他資料で参照しない**。
    - **画面遷移**: 画面一覧の遷移先 ＝ 機能要件の「完了後の動作」。同一操作の遷移先は 1 つに統一する。
    - **用語**: 同一概念は 1 表記に統一。揺れが検出されたら用語集を更新し全資料を合わせる。
13. **データモデルは概念レベル** で記述する。主要エンティティ（ENT-XXX）・関係・多重度・ステータスを持つエンティティの状態遷移（ST-XXX）を `データモデル.md` に集約する。カラム型・PK/FK・インデックス・物理テーブル名は書かない（設計フェーズの責務）。
14. **外部インターフェース一覧** を `外部インターフェース一覧.md` に作成し、メール送信・バッチ・外部システム連携を EXT-XXX で採番する。第 1 版で実装しない IF も「対象外」として明示する。「外部 IF なし」と判断する場合もその旨と根拠を残す。
15. **移行要件** を `移行要件.md` に作成し、初期データ・マスタデータ・既存データ移行を MIG-XXX で採番する。第 1 版で移行が発生しないと判断する場合も「移行なし」と明記して根拠を残す。
16. **非機能要件のベースライン**（パスワードポリシー、セッション有効時間、対象ブラウザ・デバイス、バックアップ最低頻度、障害通知方法、稼働時間帯）は TBD を本文に残さない。確定済みなら表に値を入れ、未確定なら `オープン課題.md` の `Q-NF*` へ逃がして本文からは参照のみとする。
17. **オープン課題のクローズ手順** を `オープン課題.md` の冒頭に必ず章として記載し、「設計着手前にクローズ必須の課題区分」と「設計フェーズへ持ち越して良い課題区分」を明示する。
18. **業務フロー** は成約・運送・評価のような中核フローに加え、**アカウント登録（法人重複判定）** と **論理削除→物理削除** に分岐や複数アクター間の引き継ぎがある場合は `activities/` 配下にも追加する。
18.1. **権限（認可）マトリクス** を `権限マトリクス.md` に作成し、ロール × ユースケース（UC-XXX）/リソース（ENT-XXX・SCR-XXX）の操作可否（C/R/U/D・状態遷移）とテナント越境拒否条件を一覧化する。設計フェーズの認可設計の入力とする。ロールが 1 種で認可不要なら、その旨と根拠を明記する。
18.2. **メッセージ一覧** を `メッセージ一覧.md` に作成し、エラー/警告/情報/確認/バリデーション/通知の利用者向け文言に **MSG-XXX（3 桁ゼロ埋め）** を採番する（区分 / 文言 / 表示画面 SCR-XXX / トリガ）。未確定文言は本文に断定で書かず `オープン課題.md`（Q-MSG*）へ切り出す。
18.3. **コード値定義** を `コード値定義.md` に作成し、業務区分値（ステータス・種別）の概念正典（区分名 / コード値 / 表示名 / 意味 / 関連 BR・ST）を置く。物理 enum は設計フェーズの責務。「区分値なし」の場合もその旨を明記する。
18.4. **通知・文面定義** を `通知・文面定義.md`（または `メッセージ一覧.md` に統合）に作成し、通知タイプ・送信メールの件名/本文/差込変数/関連 UC・BR・EXT を記述する。通知・メールが無い場合もその旨を明記する。
19. この skill の責務は要件定義成果物の作成までとし、設計、実装、UT、静的解析、E2E の実行やファイル作成には進まない。
20. 最後に、人手レビューと採択後に `design-from-requirements` を起動する必要があることを明記する。

## 完了条件

- `docs/requirements/` 以下に上記のファイル構成で要件定義成果物が出力されている。
- 機能ごとに個別ファイルが作成されている。
- 画面一覧に画面 ID（SCR-XXX 形式）が採番され、画面遷移図が Mermaid で描かれている。
- `ユースケース図.md` が 1 枚作成され、UC-XXX が採番されている。`概要.md` および `functional/[機能名].md` から UC-XXX で参照されている。
- 業務プロセスに分岐がある業務フローについて `activities/[フロー名].md` が作成され、ACT-XXX が採番されている。各ステップで関連 UC-XXX / BR-XXX / AC-XXX が明示されている。
- `データモデル.md` が作成され、ENT-XXX 採番・概念 ER 図（Mermaid `erDiagram`）・ステータスを持つエンティティの状態遷移（ST-XXX）が含まれている。
- `外部インターフェース一覧.md` が作成され、EXT-XXX 採番のメール送信 / バッチ / 外部システム連携が列挙されている（「外部 IF なし」と判断する場合もその旨が明記されている）。
- `移行要件.md` が作成され、MIG-XXX 採番の初期データ・マスタデータ・既存データ移行方針が記述されている（「移行なし」と判断する場合もその旨が明記されている）。
- `権限マトリクス.md` が作成され、ロール × 操作可否とテナント越境拒否条件が一覧化されている（「認可制御なし」の場合もその旨が明記）。
- `メッセージ一覧.md` が作成され、MSG-XXX 採番の利用者向け文言が整理されている（未確定文言は Q-MSG* に切り出されている）。
- `コード値定義.md` が作成されている（「区分値なし」の場合もその旨が明記）。
- `通知・文面定義.md` が作成されている、または メッセージ一覧.md に統合されている（通知・メールが無い場合もその旨が明記）。
- `非機能要件.md` のベースライン項目（パスワードポリシー / セッション / 対象ブラウザ・デバイス / バックアップ / 障害通知 / 稼働時間帯）が TBD のまま本文に残っていない。未確定項目は `オープン課題.md` の `Q-NF*` に切り出されている。
- `概要.md` にスコープ・対象外・前提が明記されている。
- `オープン課題.md` に未解決事項が一覧化され、**クローズ運用ルール章（設計着手前クローズ必須課題の区分とクローズ手順）** が含まれている。
- 後続工程は未着手であり、人手レビュー待ちであることが明記されている。
