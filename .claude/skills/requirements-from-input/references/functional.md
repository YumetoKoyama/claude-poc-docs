## 成功基準

> 実装完了の判定基準を **定量的・技術非依存** な指標で定義する。
> フレームワーク名・ライブラリ名・データベース名・API 応答時間などの実装詳細は書かない。
> 「利用者視点で何が達成されれば成功か」を記述する。

- **SC-001**: [定量指標 — 例: 利用者が 3 分以内に登録を完了できる]
- **SC-002**: [定量指標 — 例: エラー発生時に原因が判別できるメッセージが表示される]
- **SC-003**: [定量指標 — 例: 検索結果が 1 秒以内に表示される（利用者体感）]

> 成功基準は後続の非機能テスト計画（`docs/design/非機能テスト計画.md`）の検証シナリオと対応づける。

## 受け入れ条件

> 受け入れ条件は実行可能な単位で記述する。後続の単体テスト・E2E が「受け入れ条件 → 実行可能テスト」の対応を取れる粒度まで分解すること。
> 各 AC は **Given（前提）/ When（操作）/ Then（期待結果）** の形式で記述し、テスト可能性を担保する。

### 正常系

- [ ] AC-001: Given [前提] / When [操作] / Then [期待結果]

### 異常系

### 境界値・特殊条件
