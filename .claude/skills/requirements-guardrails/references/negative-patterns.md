## カテゴリA: 未確定のまま通過させてはいけない事項

### A-1: 認証・セキュリティ系

| 避けるべき記述 | 代わりに出力すべき内容 |
|--------------|----------------------|
| 「パスワードポリシーは後で決める」 | [要確認] パスワード複雑性要件（最低文字数、文字種、期限）を確認する。影響: ログイン画面仕様・バックエンド認証ロジック |
| 「ログイン失敗時はロックする」（条件なし） | [要確認] ロック条件（失敗回数・時間窓・ロック解除方法）を確認する |
| 「再設定URLを送信する」（有効期限なし） | [要確認] 再設定URLの有効期限（例: 24時間）を確認する |
| 「同時ログインの扱いは検討中」 | [要確認] 同時ログイン制御方針（許可する/後発を弾く/両方通知）を確認する |

### A-2: データ定義系

| 避けるべき記述 | 代わりに出力すべき内容 |
|--------------|----------------------|
| 「ログインIDは一意とする」（スコープ不明） | [要確認] ログインIDの一意性スコープ（システム全体か企業単位か）を確認する |
| 「金額を入力する」（単位なし） | [要確認] 金額の単位（円・千円・万円）と小数点桁数を確認する |
| 「容積を入力する」（単位なし） | [要確認] 容積の単位（m³・L等）を確認する |
| 「物品種別・トラック種別」（定義なし） | [要確認] マスタの初期値一覧と追加・変更の運用ルールを確認する |

### A-3: 業務制約系

| 避けるべき記述 | 代わりに出力すべき内容 |
|--------------|----------------------|
| 「応募締切を過ぎた案件には応募できない」（境界不明） | [要確認] 締切境界判定（締切日の23:59:59まで可/00:00:00から不可）を確認する |
| 「セット応募が競合した場合は調整する」 | [要確認] セット応募競合時の解消ルール（先着優先/発注者裁量/全取消）を確認する |
| 「最終条件を提示する」（上書き可否不明） | [要確認] 最終条件の上書き可否（提示後変更不可/合意前まで変更可）を確認する |

### A-4: 運用系

| 避けるべき記述 | 代わりに出力すべき内容 |
|--------------|----------------------|
| 「定期的にバッチを実行する」 | [要確認] バッチ実行時刻（例: 毎日2:00）と失敗時の再実行ポリシーを確認する |
| 「通知履歴を保持する」（期間なし） | [要確認] 通知の保持期間（例: 90日）と削除方針を確認する |
| 「主要ブラウザに対応する」 | [要確認] 対象ブラウザ（Chrome/Edge/Safari）と最低サポートバージョンを確認する |

## カテゴリB: 例外系・境界条件のパターン

### B-1: 競合制御

**問題のある書き方:**
```
応募者が「応募確定」を実行すると、応募が確定される。
```

**正しい書き方:**
```
応募者が「応募確定」を実行すると、応募が確定される。

[例外] 確定操作時点で応募枠が満了していた場合:
- [要確認] 「応募不可」エラーを返すか、先着順で受け付けるかを確認する

[例外] セット応募の確定中に他の応募が枠を埋めた場合:
- [要確認] セット全体を不成立とするか、成立分のみ確定するかを確認する
```

### B-2: 取消・差戻し

**必ず記述すべき例外ケース:**
- 運送開始報告後に「開始報告を取り消せるか」
- 完了報告後に差戻しが発生した場合のステータス遷移
- 合意フェーズ中に一方が離脱した場合の扱い
- 合意フェーズ中に他社が同一案件で先に成約した場合

### B-3: 削除中データの表示

**問題のある書き方:**
```
削除された案件は表示しない。
```

**正しい書き方:**
```
削除操作から24時間は論理削除状態で保持する。

[要確認] 論理削除中（24時間保持中）の案件の画面表示方針を確認する:
- 選択肢A: 発注者のみ「削除済み（復元可）」として表示
- 選択肢B: 全ユーザーから非表示
- 選択肢C: 関連する応募者にのみ「案件が削除されました」として表示
```

## カテゴリC: 文書間不整合パターン

### C-1: ステータス定義の揺れ

**検出方法:**
1. 用語集のステータス定義一覧を正とする
2. 業務ルール・画面仕様・フロー図に登場するステータス名を抽出
3. 用語集に存在しないステータスが他資料に現れていないか確認
4. 「第1版では使用しない」と定義されたステータスが他資料で参照されていないか確認

**特に注意すべきパターン:**
- 「キャンセル」状態が用語集では第1版対象外なのに、フロー図やルールで参照されている
- 「運送中」という独立ステータスが一部資料にのみ存在する

### C-2: 画面遷移の揺れ

**検出方法:**
- 画面一覧の「遷移先」列と、機能要件の「完了後の動作」記述を突き合わせる
- 同一操作について遷移先が1つに統一されているか確認

**特に注意すべきパターン:**
- アカウント登録完了後: 画面一覧と機能要件で遷移先が異なる場合
  - 「ログイン画面へ遷移」と「ダッシュボードへ自動ログイン」は実装が根本的に異なる

### C-3: 用語の揺れ

**統一すべき用語ペア例:**
| 揺れのある表現 | 統一すべき用語 |
|--------------|--------------|
| 成約 / 契約成立 / 確定 / 合意完了 | 1つに統一し用語集に定義する |
| 応募者 / キャリア / 配送業者 | 用語集の定義に合わせて統一する |
| 発注者 / 荷主 / クライアント | 用語集の定義に合わせて統一する |

## カテゴリD: 要件定義書への設計情報混入パターン

### D-1: 技術前提の混入

**NG（設計情報）:**
```
認証にはSpring Securityを使用し、JWTトークンで状態を管理する。
```

**OK（要件情報）:**
```
認証方式: セッションベースまたはトークンベース（方式は設計工程で決定）
要件: ログイン状態は一定時間操作がない場合タイムアウトすること
[要確認] タイムアウト時間（推奨: 30分）を確認する
```

**NG（設計情報）:**
```
フロントエンドはReactで実装し、状態管理にはReduxを使用する。
```

**OK（要件情報）:**
```
画面はWebブラウザで動作するSPAとする。
対象ブラウザ: [要確認]
```

### D-2: ブランド・UI指針の混在

**問題:** ブランドガイドライン・UIライブラリ選定・多言語・トーン&ボイスが
業務要件と同一の優先度で記述されている。

**対応:** これらはUI/UX検討事項セクションまたは別文書として分離し、
要件確定のクリティカルパスから外す。

**分離後の記述例:**
```markdown
## 業務要件（確定必須）
- ...

## UI/UX検討事項（設計工程で詳細化）
- ブランドカラー・ロゴ指針
- UIコンポーネントライブラリの選定
- 多言語対応の有無
- トーン&ボイスガイドライン
```
