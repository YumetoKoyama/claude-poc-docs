# 設計書から UI 設計ブリーフを作成する

> **パス解決（マルチリポジトリ対応）**: 本スキル内の `docs/requirements/`・`docs/design/`・`docs/test/` は **docs リポジトリ（claude-poc-docs）ルート相対**のパスを指す。
> - docs リポジトリをカレントとして実行している場合: そのまま使う。
> - 親アンブレラ（claude-poc-rules）から実行している場合（カレント直下に `claude-poc-docs/` が存在する場合）: これらすべてのパスに `claude-poc-docs/` を前置して読み書きする。
> - CI（子リポジトリ単体のチェックアウト）で docs リポジトリが存在しない場合: workflow が追加チェックアウトした docs のパスを使う。それも無い場合は Issue 本文に埋め込まれた設計情報を入力とし、原本の参照が必要なら中断して人間に確認する。

この skill の入力は **採択済みの設計書**（`docs/design/`）と、要件定義の **`docs/requirements/ブランドガイドライン.md`**（任意）です。設計書が存在しない場合、または採択済みでない場合は中断し、先に `design-from-requirements` の実行とレビュー採択を依頼すること。

> **採択ゲート（必須）**: 採択は「設計書の PR を人間がレビューし `main` へマージすること」で行う（branch protection で直 push 禁止が前提）。開始前に、入力の設計書（`docs/design/`）が `main` に存在することを確認する（ローカルでは `git log origin/main -- docs/design/` 等で確認）。未マージなら中断し、人手レビュー・マージを依頼する。

設計入力: $ARGUMENTS（省略時は `docs/design/` 配下）

## 設計方針（重要）

Claude Design の Export 物は **画面単位ではなくカテゴリ単位 + 共有資産**（共通プリミティブ / トークン / シェル）で構成されることが実証されている。したがって UI ブリーフも同様に「共通 + 画面別」の2層構造で作成し、Claude Design へは:

1. **共通ブリーフ（`_共通.md`）を最初に投入** してデザインシステム・共通コンポーネント・ロール・トーンを確立
2. 続いて **画面別ブリーフ（`[scr-id]-*.md`）をまとめて添付**

する流れを取る。画面別ブリーフは共通ブリーフからの **差分のみ** を記述し、DS や共通コンポーネントの再掲は避ける。

## 役割

採択済み設計書を読み、以下の2層のブリーフ md を出力する:

- **共通ブリーフ**: 全画面横断の DS・共通コンポーネント・ロール・トーン & ボイス・レスポンシブ方針・状態表現規約
- **画面別ブリーフ**: 画面ごとの Goal / Audience / Content / Layout / States / Interaction の **差分**

このスキルは設計書を **書き換えない**。設計書はあくまで Source of Truth として参照する。

## 成果物

- `docs/design/ui-design/brief/_共通.md` — 全画面共通のブリーフ（DS・共通コンポーネント・ロール・トーン・状態規約・レスポンシブ方針）
- `docs/design/ui-design/brief/[scr-id]-[画面名].md` — 画面ごとに 1 ファイル（`SCR-XXX-画面名.md` 形式の日本語名、ファイル冒頭に画面 ID を記載、共通ブリーフからの差分のみ）
- `docs/design/ui-design/brief/README.md` — Claude Design への投入手順（共通 → 画面別の順序）と全画面の索引

> 既存の `docs/design/screens/` には触れない。UI ブリーフはここに **増分** として置き、設計書と Claude Design 入力の間を埋める層と位置付ける。

## 指示

### 1. 入力把握

1. `docs/design/概要.md` と `docs/design/screens/画面遷移.md` を読み、全体スコープと画面遷移を把握する。
2. `docs/requirements/ブランドガイドライン.md` を読み、DS の方針・ペルソナ・トーン & ボイス・レスポンシブ方針を把握する。存在しない場合はスキップ可だが、README にその旨を記載する。
3. `docs/design/screens/` 配下の `*.md` を列挙し、全画面の一覧を作成する。

### 2. 共通ブリーフ（`_共通.md`）の生成

`docs/design/ui-design/brief/_共通.md` を作成する。最低限、次のセクションを必ず埋める。記述は **全画面に共通する事項のみ** とし、特定画面固有の事項は画面別ブリーフに逃がす。

- **プロダクト概要**: 1〜2 段落で何のシステムか・主要ユーザー種別（ロール）
- **ロール / チャネル**: 役割（例: SHIPPER / CARRIER / 共通未認証）と、各役割が触れる画面群
- **トーン & ボイス**: 言語（日本語固定など）、敬体/常体、絵文字使用可否、ボタンの命名規則
- **Design System**:
  - カラーパレット（プライマリ / セカンダリ / アクセント / 警告 / 背景 / テキスト系の論理名と用途）
  - タイポグラフィ（フォントファミリ・スケール・行高・用途別マッピング）
  - スペーシング・角丸・シャドウ・フォーカスリングのスケール
  - アイコンライブラリ（例: Lucide outline, 1.5px stroke）
- **共通コンポーネント**: AppHeader / SideNav / Button / Field / Modal / Table / StatusBadge / Tag / Card / Panel / Pagination 等の振る舞いと寸法規約
- **状態表現規約（States）**: empty / loading / error / 権限不足 の表示方針と既定コピー
- **レスポンシブ方針**: 想定デバイスとブレークポイント、最大幅、グリッド
- **アクセシビリティ**: 準拠レベル、キーボード操作、フォーカスリング規約
- **アニメーション / モーション**: 既定 easing・duration、禁止事項

各セクションの末尾に `参照:` で出典（設計 md または要件 md のパス + 節名）を必ず付ける。設計書に書かれていない仕様は新たに決めない。

### 3. 画面別ブリーフ（`[scr-id]-[画面名].md`）の生成

`docs/design/screens/` 配下の全画面について、`docs/design/ui-design/brief/[scr-id]-[画面名].md` を作成する（ファイル名は対応する設計 md と同じ日本語名で揃える）。各画面別ブリーフは [ui-brief-template.md](ui-brief-template.md) の構造に従い、**共通ブリーフからの差分のみ** を記述する。最低限、次のセクションを必ず埋める:

- **Goal**: この画面の一次目的・二次目的
- **Audience（差分）**: 共通ブリーフの想定ロールから絞られる場合のみ記述（例: SHIPPER のみ）
- **Content**: 表示する主要情報・固有コピー・空データ時のメッセージ（設計 md の「出力・表示内容」を再構成）
- **Layout**: ワイヤフレーム的な領域分割と情報階層（共通シェルの上に画面固有のグリッドや Panel 配置）
- **States（差分）**: 共通の状態規約から逸脱する/追加で必要な状態のみ
- **Interaction**: 画面固有のボタン挙動・モーダル遷移・キーボードショートカット
- **画面固有のバリデーション**: 入力フィールド単位の制約（設計 md「バリデーションメッセージ」から抽出）
- **画面遷移**: この画面からの遷移先（操作 → 遷移先画面 ID）

DS・共通コンポーネント・トーンは **再掲しない**（共通ブリーフを参照と書くに留める）。Claude Design への投入時に共通ブリーフがコンテキストに乗っているため、重複は context window の浪費になる。

### 4. トレーサビリティ

- 各セクションの記述は **必ず設計書または要件定義のどこから来たかを `参照:` で明記**する。
- 設計書側の不足は画面別ブリーフの「未確定事項」節に列挙し、必要なら共通ブリーフ末尾の「未確定事項（全体）」節にも追記する。

### 5. Claude Design 投入用プロンプト

- 共通ブリーフ末尾に「Claude Design 投入用プロンプト（共通投入時）」セクションを設け、デザインシステム確立を依頼する 1 段落のテキストを置く。
- 画面別ブリーフには **個別のプロンプトは不要**（共通投入後にブリーフ md を添付するだけで Claude Design が読む）。ただし画面固有の強調事項があれば「強調したい点」節に箇条書きで置く。

### 6. README.md（索引）

`docs/design/ui-design/brief/README.md` には次を含める:

- ファイル構成図（`_共通.md` + 画面別 md + README.md）
- 画面 ID → 画面別ブリーフ md のリンク一覧
- **Claude Design 投入手順（重要・順序固定）**:
  1. claude.ai/design でプロジェクトを新規作成
  2. **最初に `_共通.md` を添付** し、共通ブリーフ末尾の「投入用プロンプト」を貼ってデザインシステムを確立
  3. Claude Design が DS を受け入れ、必要な確認応答が終わったら、**画面別ブリーフを一括（または役割/カテゴリ単位）で添付** して各画面の生成を依頼
  4. 生成結果を確認し、必要に応じて対話で調整
  5. Export → Handoff bundle を取得し、**Export 構造そのまま** `docs/design/ui-design/handoff/` 配下に格納（このディレクトリ作成と格納は人手で実施）
- **`docs/design/ui-design/handoff/` の格納ルール**: Export の `README.md` / `prototype/` / `tokens/` を分割せずそのまま置く。画面単位 (`[scr-id]/`) で切り分けてはいけない（共通プリミティブ・シェル・トークンは横断的共有資産であり画面単位に分割不能なため）。
- デザインシステムの取り扱い: Claude Design 側の組織デザインシステムを優先するか、`_共通.md` で上書きするかの判断指針

### 7. 既存ファイル・ブランドガイドラインの扱い

- `ブランドガイドライン.md` が存在しない場合は、`_共通.md` の Design System / Audience セクションを「未指定（Claude Design 側のデフォルトに従う）」と明示し、`docs/requirements/オープン課題.md` に追記すべき項目を README に列挙する。
- 既にブリーフファイルが存在する場合は該当ファイルのみ更新し、無関係なファイルは書き換えない。
- このスキルは UI ブリーフの作成までを責務とし、Claude Design の実行・スクリーンショットの取得・実装・テストには進まない。

## 完了条件

- `docs/design/ui-design/brief/_共通.md` が存在し、DS・共通コンポーネント・ロール・トーン・状態規約・レスポンシブ方針が網羅されている。
- `docs/design/ui-design/brief/` 配下に、画面 ID ごとに 1 ファイルずつ画面別ブリーフ md が出力されている。
- 各画面別ブリーフは共通ブリーフからの差分のみで構成され、DS・共通コンポーネントの再掲がない。
- 各セクションに `参照:` で設計 md・要件 md の出典が明示されている。
- `docs/design/ui-design/brief/README.md` に索引と Claude Design 投入手順（共通 → 画面別の順序）、および ui-design/handoff の格納ルール（Export 構造そのまま）が記載されている。
- 設計書本体（`docs/design/screens/` 等）は書き換えられていない。

## 注意事項

- このスキルは Claude Design を **代替しない**。Claude Design 側でデザインシステム適用・対話的調整・ビジュアル生成を行い、本スキルはその入力品質を担保するだけ。
- UI ブリーフは設計フェーズの **採択後** にだけ作成する。設計が確定していない段階で UI を作っても、設計変更で破棄になる。
- `_共通.md` を画面別ブリーフより先に投入する順序は守ること。逆順だと Claude Design は画面ごとに DS を再発明してしまい、画面間の一貫性が崩れる。
- フィードバックループ（review-ui-design/brief / fix-ui-design/brief）は現状未整備。必要になった段階で `*-loop` パターンに合わせて拡張する。
