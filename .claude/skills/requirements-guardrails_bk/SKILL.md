# 要件定義書 ガードレールスキル

このスキルは、AIが要件定義書を生成・補完する際に、過去の評価で判明した典型的な欠陥パターンを
出力しないためのネガティブプロンプトとチェックリストを提供する。

> **汎用化に関する注記**: 本スキルおよび本文中の具体例（「セット応募」「荷主」「運送開始・完了報告」「合意フェーズ中に他社が先に成約」「アカウント登録の法人重複判定」など）は、検証に用いた **運送マッチングという参考ドメイン固有の実例** である。別ドメインに適用する場合は、これらを抽象パターン（「同時操作の競合制御」「中間状態の表示方針」「一意性スコープの確定」等）として読み替えること。ドメイン固有の必須フローはプロジェクト個別の追加指示として別管理し、本スキルの抽象パターン部分のみを横断的に再利用する。

## MUST NOT（絶対に出力してはいけないパターン）

以下に該当する内容を要件定義書に含めてはならない。
該当する判断が必要な場合は、出力せずに「要確認事項」として明示的に未解決課題へ分類する。

### カテゴリA: 未確定のまま通過させてはいけない事項

```
❌ パスワードポリシー・アカウントロック条件・セッション有効期限を「後で決める」として省略する
❌ ログインIDの一意性スコープ（システム全体か企業単位か）を曖昧なまま記述する
❌ 金額・容積・数量の単位を明記せずに仕様として記述する
❌ 応募締切の境界時刻判定（00:00は当日含む/翌日扱い等）を未定義にする
❌ バッチ処理の実行時刻・失敗時の扱いを「TBD」「後で決める」のまま残す
❌ 通知の保持期間・削除ポリシーを未定義にする
❌ 対象ブラウザ・対象デバイスを未定義のまま非機能要件を記述する
❌ バックアップ取得頻度と復旧目標時間（RTO/RPO）を記述しない
❌ 障害通知の責任範囲・連絡経路を記述しない
❌ ログ保管期間を記述しない
```

### カテゴリB: 例外系・境界条件を正常系のみで閉じてはいけない

```
❌ 複数人が同時に同一応募枠に応募する競合ケースの制御方針を記述しない
❌ セット応募確定中に応募枠が埋まった場合の処理を記述しない
❌ 合意フェーズ中に他社が先に成約した場合の扱いを記述しない
❌ 完了報告後の差戻し要否を記述しない
❌ 運送開始・完了報告の取消可否を記述しない
❌ 論理削除中データ（削除後24時間等の保持期間中）の画面表示方針を記述しない
❌ 物理削除後の関連データ（応募・通知・連絡履歴など）の扱いを記述しない
```

### カテゴリC: 文書間不整合を生んではいけないパターン

```
❌ ステータス種別（例:「運送中」という状態が存在するかどうか）を資料間で不一致にする
❌ 画面遷移先を画面一覧と機能要件とで異なる記述にする
   例: 登録完了後→ログイン画面 と 登録完了後→ダッシュボード が混在する
❌ 「第1版では使用しない」と定義したステータス（例:キャンセル）を
   別の資料でエラー条件・不成立条件として参照する
❌ 同一概念に複数の表記を混在させる
   例: 「成約」「契約成立」「確定」が同じ意味で混在
❌ 業務ルール文書と画面仕様で同一項目の条件が異なる
```

### カテゴリD: 要件定義書に混入してはいけない設計・実装情報

```
❌ フレームワーク名・ライブラリ名を要件として記述する
   例: "Spring Securityで認証を実装する" → NG（要件ではなく設計方針）
   例: "Reactでフロントエンドを構築する" → NG（要件ではなく技術選定）
❌ DBスキーマの列名・型定義を要件定義書に記述する
❌ APIエンドポイントのパス設計を要件として記述する
❌ ブランドガイドライン・UIライブラリ・多言語対応・トーン&ボイスを
   業務要件と同じ優先度・粒度で業務要件一覧に混在させる
   （これらはUI/UX検討事項として別セクションまたは別文書で管理する）
```

## 未解決事項の扱い方（代替出力パターン）

MUST NOTに該当する判断が必要な場合、要件として断定せず以下の形式で出力する。

```markdown
> **[要確認]** {確認が必要な内容を1文で記述}
> - 選択肢A: {パターン1}
> - 選択肢B: {パターン2}
> - 影響範囲: {この決定が影響する機能・画面・ステータス遷移}
> - 確認期限の推奨: {設計着手前 / DB設計前 / UI実装前 など}
```

未解決事項はオープン課題一覧に転記し、要件本文では参照リンクを残すこと。
「後で決める」「TBD」だけで置くことは禁止。

## データ・IF定義の最低ラインチェックリスト

要件定義書を出力する前に、以下の項目が定義済みであることを確認する。
未定義のものは要件として断定せず「要確認」として扱う。

### エンティティ定義
- [ ] 主要エンティティ（案件・応募・セット応募・通知・評価など）の識別子と保持単位が定義されている
- [ ] 論理削除の対象エンティティと保持期間が定義されている
- [ ] 物理削除後に関連データをどう扱うかが定義されている

### 外部IF一覧
- [ ] メール送信の実行主体（バッチ/イベント駆動）が定義されている
- [ ] 通知の実行タイミングと配信方法が定義されている
- [ ] バッチ処理の一覧・実行スケジュール・失敗時の扱いが定義されている

### 監査・追跡
- [ ] 最低限ログを残す操作の一覧が定義されている
- [ ] ログ保管期間が定義されている

## 非機能要件の最低ラインチェックリスト

以下の項目が「方針」ではなく「要求値」として記述されていることを確認する。

| 項目 | NG例（方針止まり） | OK例（要求値） |
|------|------------------|--------------|
| 対象ブラウザ | 「主要ブラウザに対応する」 | 「Chrome最新版、Edge最新版、Safari最新版（iOS含む）」 |
| 対象デバイス | 「スマートフォン対応」 | 「スマートフォン（iOS 16以上、Android 13以上）縦向き」 |
| 可用性 | 「高可用性を目指す」 | 「稼働時間: 平日9:00-22:00、計画停止は事前通知」 |
| バックアップ | 「定期的にバックアップを取る」 | 「1日1回、復旧目標: 前日データまで、RTO 4時間以内」 |
| 性能 | 「快適に動作する」 | 「一覧画面: 3秒以内（同時接続50ユーザー時）」 |
| 障害通知 | 「障害時は対応する」 | 「P1障害: 15分以内に担当者へSlack通知、エスカレーション先: XX」 |

## 整合性チェック（文書間クロスリファレンス）

要件定義書の各資料を出力・更新するとき、以下の整合を確認する。

### ステータス遷移の整合
- 用語集に定義されたステータス一覧 ＝ ステータス遷移図に登場するステータス
- ステータス遷移図 ＝ 業務ルールの遷移条件
- 業務ルールの遷移条件 ＝ 画面の操作・通知条件

特に注意: **第1版対象外と定義したステータスは他の資料でも参照してはならない**

### 画面遷移の整合
- 画面一覧の遷移先リスト ＝ 機能要件の「完了後の動作」記述
- 同一の操作完了後の遷移先が1つに統一されていること

### 用語の整合
- 同一概念は1つの表記に統一すること
- 表記ゆれが検出された場合は用語集を更新し、全資料で統一する

## 良い点（変更・追記不要なパターン）

以下は既存の要件定義書で良く機能している点であり、**変更や追記をしてはいけない**。

```
✅ 業務スコープ（対象・対象外）の明示
✅ 機能資料ごとの入力・出力・受け入れ条件・未解決事項の構成
✅ UC / BR / AC / SCR / ACT の相互参照構造
✅ オープン課題の別紙管理（未確定事項を埋もれさせない構成）
✅ 背景・目的・前提条件・制約の整理
```

これらの構造・フォーマットは維持しながら、不足している内容を補完すること。
