# 権限マトリクス

> ID 凡例: [docs/凡例.md](../凡例.md) 参照

## 1. 権限方針サマリ

- 第 1 版はテナント内「同権限のみ」の単純なマルチユーザーであり、テナント内でのロール分離・承認フローは存在しない（BR-003）。管理者／一般等のロール分離は次期開発で対応する。
- 一方で、**テナント（企業アカウント）単位のデータ境界は必須**である。配送依頼企業と運送会社という 2 つのアクター種別が存在し、それぞれ「自社（自テナント）が当事者のデータ」にのみアクセスできる。本表ではこの 2 アクター種別を便宜上「ロール」として扱う。
- 認可制御が存在しない（BR-003）のはテナント**内**のユーザー間のみであり、テナント**間**の越境アクセスは常に拒否される。

## 2. ロール一覧

| ロール | 対応するテナント種別 | 概要 |
|-------|------------------|------|
| 配送依頼企業ユーザー | ENT-001（企業種別＝配送依頼企業） | 案件登録・審査・交渉・合意・評価を行う |
| 運送会社ユーザー | ENT-001（企業種別＝運送会社） | 応募・交渉・運送実施報告・評価を行う |

## 5. テナント越境拒否条件

- 自テナントが当事者でない案件・応募・連絡メッセージ・成約スナップショット・評価・通知への直接アクセス（URL 直打ち等を含む）は**すべて拒否**する。
- ただし「募集中」ステータスの案件（ENT-003）は業務上の性質として、すべての運送会社テナントに公開される（マッチング機能の前提であり越境には当たらない）。交渉中以降は当該案件に応募した運送会社のみが自社の応募状況を参照できる。
- 競合表示（他社の応募金額）は匿名化（「A社」等）した上で開示する（BR-021）。会社名等の識別情報は非開示のままとする。
- テナント越境時の HTTP レスポンスコード等の技術的な統一方針（404 か 403 か）は設計フェーズ（セキュリティ設計）の責務とし、本書では業務要件としての「越境禁止」のみを定義する。

## 6. 開かれた論点

現時点で権限マトリクスに起因する未解決事項はない。第 1 版が「同権限のみ」であるため、将来のロール分離（管理者／一般）導入時は本マトリクスの改訂が必要になる（将来課題）。
