以下が、「第1版・配送マッチングWebアプリ」の要件まとめです。
ユーザー・アカウント
1-1. ユーザー種別
配送依頼企業（依頼側／旧「荷主」）
※モデルとしては大手物流企業（例：ヤマト）を想定するが、利用者を特定企業に固定しない。
あくまで「配送を依頼する企業」全般が対象であり、依頼側を固定するとマッチングが成立しないため、汎用的に扱う。
運送会社（運び手）
1-2. 企業アカウントとユーザーの関係
1企業に対して1つの「企業アカウント（テナント）」を発行する。
1つの企業アカウントの配下に、同権限の利用ユーザーを複数（N人）作成できる。
利用単位は営業所等を想定（例：ヤマト ○○営業所、ヤマト △△営業所）。
セキュリティの観点から、営業所内で1アカウントを使い回す運用は推奨しないため、
　ユーザー単位でログインIDを発行する仕組みを必須とする。
第1版では「同権限のみ」の単純なマルチユーザー（権限分離・承認フロー無し）。
　管理者／一般などのロール分離は次期開発で対応。

応募モデル
3-1. 応募単位
1応募 ＝ 1案件に対する1社の1提案
1社が同じ案件へ複数提案することはしない（複数便が必要な場合は条件内で表現）
3-2. セット応募（複数同時応募可）
1社が複数案件をまとめて「セット」で応募できる。
　例：案件A・B・Cが同時に受注できる場合に限り運ぶ、という受け方。
セット応募の対象範囲：
　同一の配送依頼企業（テナント）が登録した案件のみセット可とする。
　異なる配送依頼企業の案件を1つのセットに混在させることはできない
　（合意主体を1社に統一するための制約。複数依頼企業をまたぐセットは将来拡張）。
セット応募の成立条件は「セット内すべての案件で合意成立」（オール・オア・ナッシング）。
　セット内のいずれか1件でも他社に成約・キャンセルされた場合、当該セット応募は自動的に不成立となる。
セット内の各案件は、個別案件としても引き続き他社の応募を受け付ける（セット応募は1つの提案として並列に存在する）。
配送依頼企業側の詳細画面では、応募が「単体」か「セット（連動案件あり）」かを明示し、
　セット応募の場合は連動している他案件も併せて確認できるようにする。
合意フェーズ（後述）でも、セット応募はセット単位で「最終条件提示」「合意」を行う。
　セット応募に対する合意は、配送依頼企業側で「セット一括」での承認操作とする
　（案件ごとに個別合意するのではなく、セット全体で同時に合意成立扱い）。
3-3. 応募上限・締切
1案件あたり応募上限：20社まで（先着順）
　セット応募の場合も、セット内の各案件それぞれで応募枠を1つ消費する。
自動締切：from時刻の2時間前に応募不可とする
将来、配送依頼企業が任意の締切を設定できるよう拡張予定

応募

希望金額に同意 or 独自金額を入力

交渉
双方、連絡機能で金額・条件をやり取り

相手が「合意する」ボタン
この2ステップで合意成立とする
成約処理（ステータス：成約済）
合意時点で次の内容をスナップショットとして保存（以後編集不可）
合意金額
from/to の場所・日時
トラック種別
双方の企業情報・担当者情報
合意時点のメッセージ情報 等
他応募者には「他社に決定しました」通知を送付し、その応募をクローズ
　セット応募がクローズされる場合、同セットに含まれていた他案件側の当該応募も連動してクローズし、
　応募していた運送会社に「連動セットが不成立になりました」旨を通知する。
成約後のキャンセル機能は第1版では提供しない
成約後のフロー・ステータス遷移
成約済 → 運送中
運送会社が「運送開始」等を報告した時点で遷移
運送中 → 完了
運送会社が「完了報告」
配送依頼企業が内容確認し「完了確認」ボタン → ステータス「完了」
完了 → 評価済
双方が★評価を登録完了 → ステータス「評価済」
評価は一度だけ・編集不可（将来変更可能にする予定）
編集制限
ステータスが「成約済」「運送中」「完了」「評価済」の案件・応募は、
案件内容・応募内容・金額の更新／削除不可
