# 要件サマリー

> ID 凡例: [docs/凡例.md](../凡例.md) 参照

## 1. 背景と目的

配送を依頼したい企業（配送依頼企業）と、実際に荷物を運ぶ運送会社とを結び付ける「配送マッチング Web アプリ」の第 1 版を構築する。想定モデルは大手物流企業（例: ヤマト）だが、依頼側の業態を特定企業に固定すると汎用的なマッチングが成立しないため、「配送を依頼する企業」全般を対象とする汎用プラットフォームとして設計する。

目的は、案件（積荷）の登録から、運送会社による応募、金額・条件交渉、合意・成約、運送実施、完了確認、相互評価までの一連の業務プロセスを Web 上で完結させることである。

## 2. スコープ

### 対象

- 配送依頼企業・運送会社それぞれの企業アカウント（テナント）登録、ユーザー追加、ログイン認証
- 案件登録・一覧・詳細閲覧・編集・削除（募集中・交渉中のみ）
- 運送会社による応募（単体・セット応募）
- 連絡機能によるメッセージ・金額交渉
- 最終条件提示・合意による成約処理（スナップショット保存含む）
- 運送開始報告・完了報告・完了確認によるステータス管理
- 双方の★評価登録
- 通知（画面内アイコン・一覧）
- 取引履歴の閲覧
- 案件の論理削除・物理削除（夜間バッチ）

### 対象外（やらないこと）

- 決済機能（金額のやり取りは条件交渉のみで、実際の決済処理は行わない）
- 成約後のキャンセル機能・ペナルティ機能
- 管理者／一般等のロール分離・承認フロー（第 1 版は同権限のみ）
- 二要素認証・アカウント承認フロー
- 運送業許可番号・保有トラック詳細等の登録項目
- リアルタイム Push 通知（リロードによる最新化のみ）
- 詳細な操作ログの個別確認機能
- 本格的な問い合わせフォーム（第 1 版は「工事中」画面等での案内に留める）
- 利用停止申請機能

## 3. 主要ステークホルダと利用者像

| 区分 | 役割 | 主要な関心事 |
|------|------|------------|
| 配送依頼企業ユーザー | 案件を登録し、運送会社からの応募を審査・交渉・合意する | 希望条件に合う運送会社と早く安全に合意したい |
| 運送会社ユーザー | 募集中案件を検索し、応募・交渉・運送実施・報告を行う | 条件の良い案件を見つけて確実に受注し、代金を回収したい |

## 5. 前提条件

- 利用企業はインターネットに接続された PC・モバイル環境から Web ブラウザでアクセスする（対象ブラウザ・デバイスは `非機能要件.md` を参照）。
- 配送依頼企業・運送会社の双方が、企業単位でテナントを保有し、その配下にユーザーを作成する運用を理解している。
- 第 1 版は決済機能を持たないため、金額のやり取りは業務上の合意形成（条件交渉）に留まる。

## 6. リスク・制約

- 支払方法・物品種別・希望トラック種別などのマスタ値は確定済み（`オープン課題.md` Q-DM1・Q-DM2、`コード値定義.md` 参照）。
- パスワードポリシー・セッション有効時間・対象ブラウザ等の非機能ベースラインは設計着手前クローズ必須の課題区分としてすべて確定済み（`オープン課題.md` Q-NF2〜Q-NF10、`非機能要件.md` 参照）。Q-NF1（ログイン ID の一意性スコープ）もクローズ必須の対象外だが人手確認済み（ログインIDはテナント内一意、登録メールアドレスはシステム全体で一意の独立フィールド）。
- セット応募・編集制限まわりの境界条件（応募締切の境界時刻、最終条件の再提示可否等）は確定済み（`業務ルール.md` BR-010・BR-012、`オープン課題.md` Q-J1・Q-J2・Q-J3 参照）。
