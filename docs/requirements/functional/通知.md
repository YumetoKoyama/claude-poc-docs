# 機能要件: 通知

> ID 凡例: [docs/凡例.md](../../凡例.md) 参照

## 機能概要

- 機能 ID: —（関連ユースケース UC-021 を参照）
- 利用者: 配送依頼企業ユーザー、運送会社ユーザー
- 目的: 成約・応募・メッセージ・ステータス更新等のイベントをアプリ内通知として利用者に伝え、未読管理を行う
- 関連画面（[画面 ID](../画面一覧.md)）: SCR-012（通知一覧画面）
- 関連ユースケース（[UC-XXX](../ユースケース図.md)）: UC-021（通知確認）
- 関連業務フロー（ACT-XXX）: なし（ACT-001・ACT-002 等の各業務フローから発生するイベントを横断的に扱う機能のため専用の業務アクティビティ図は作成しない）
- 関連業務ルール（[業務ルール ID](../業務ルール.md)）: BR-014（他応募者クローズ通知）、BR-015（セット応募クローズの連動）、BR-024（通知未読数の表示上限）、BR-025（通知種別の表示区分）

> 参照: 画面 ID は [画面一覧.md](../画面一覧.md)、ユースケース ID は [ユースケース図.md](../ユースケース図.md)、業務ルール ID は [業務ルール.md](../業務ルール.md)を参照。

## ユーザーストーリー

### ストーリー 1 — 通知アイコンでの未読確認（優先度: P2）

ログイン中の利用者として、画面右上の通知アイコンで未読件数を一目で確認したい。なぜなら、重要なイベント（成約等）を見逃したくないため。

**独立テスト**: 未読通知が発生した状態でログインすると、通知アイコンに未読件数（上限 9+）が表示されることを確認する。

---

### ストーリー 2 — 通知一覧から該当案件への遷移（優先度: P2）

利用者として、通知一覧から通知をクリックして該当案件の詳細へ直接移動したい。なぜなら、通知内容を確認した流れですぐに対応したいため。

**独立テスト**: 通知をクリックすると該当案件の詳細画面へ遷移し、当該通知が既読になり未読数が減算されることを確認する。

## 成功基準

- **SC-001**: 未読件数は 9 件を超えると「9+」と表示される（BR-024）。
- **SC-002**: 「成約」通知は他の通知種別と明確に区別して表示される（BR-025）。
- **SC-003**: 通知を開くとその場で既読になり、未読数カウンタに即座に反映される。

| 項目名 | 型 / 形式 | 説明 |

## 受け入れ条件

### 正常系

- [ ] AC-001: Given 未読通知が存在する状態 / When 通知アイコンを確認する / Then 未読件数が表示される（9 件超は「9+」、BR-024）
- [ ] AC-002: Given 通知一覧を開いた状態 / When いずれかの通知をクリックする / Then 該当案件詳細へ遷移し、当該通知が既読になり未読数が減算される

### 異常系

- [ ] AC-101: Given 関連案件が既に物理削除されている通知 / When 通知をクリックする / Then 遷移先が存在しない旨のエラー表示（MSG-020 等）を行う

### 境界値・特殊条件

- [ ] AC-201: Given 未読件数が 9 件を超える状態 / When 通知アイコンを確認する / Then 「9+」と表示される（BR-024 の上限表示）

### 権限境界（該当する場合）

- [ ] AC-301: Given 自社宛てではない通知 / When 一覧取得を試みる / Then 表示されない（自社宛て通知のみ表示）

### エッジケース

- AC-401: 「成約」以外の通知（応募・メッセージ・ステータス更新等）はすべて「新着メッセージがあります」として一律表示する（BR-025）。個別の詳細文言は `メッセージ一覧.md` を参照。

## 前提条件（Assumptions）

- リアルタイム Push 通知は行わず、画面リロード・再訪問時に最新の未読状態を反映する。
- 通知の保持期間は `非機能要件.md` のデータ保持方針（Q-NF10）に従う。
