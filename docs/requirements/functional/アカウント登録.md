# 機能要件: アカウント登録

> ID 凡例: [docs/凡例.md](../../凡例.md) 参照

## 機能概要

- 機能 ID: —（関連ユースケース UC-001〜UC-003 を参照。機能単位の専用 ID 体系は導入しない）
- 利用者: 配送依頼企業ユーザー、運送会社ユーザー
- 目的: 企業単位の企業アカウント（テナント）を発行し、テナント配下にユーザーを作成してログイン可能な状態にする

- 関連業務ルール（[業務ルール ID](../業務ルール.md)）: BR-001（テナント発行単位）、BR-002（ユーザー単位ログイン ID 必須）、BR-003（第 1 版の権限方針）

## ユーザーストーリー

### ストーリー 1 — 企業アカウントの新規登録（優先度: P1）

配送依頼企業または運送会社の担当者として、自社の企業アカウント（テナント）と自分自身の初回ユーザーをまとめて登録したい。なぜなら、登録が完了しないと案件登録・応募等の業務機能を一切利用できないため。

**独立テスト**: 企業共通項目・初回ユーザー項目を入力し登録操作を行うと、テナントと初回ユーザーが作成され、ログイン画面から発行した ID でログインできることを確認する。

---

### ストーリー 2 — テナント配下へのユーザー追加（優先度: P2）

既に企業アカウントを持つ担当者として、同じ営業所・部署の同僚のために新しいログイン ID を発行したい。なぜなら、営業所内でアカウントを使い回す運用を避け、利用者ごとに操作履歴を分離したいため。

**独立テスト**: ログイン済みユーザーがユーザー追加操作を行うと、同一テナント配下に新しいユーザー（ログイン ID）が作成され、そのユーザーで独立してログインできることを確認する。

---

### ストーリー 3 — 法人重複の検出（優先度: P1）

システムとして、同一法人が誤って複数の企業アカウントを作成してしまうことを防ぎたい。なぜなら、テナントが分裂すると案件・応募データが分散し、後続の業務判断（自他判定等）が破綻するため。

**独立テスト**: 既存テナントと重複する法人情報で登録を試みると、登録がブロックされ重複エラーが表示されることを確認する。

## 成功基準

- **SC-001**: 新規利用者が企業共通項目・初回ユーザー項目の入力から登録完了まで一度の画面遷移フロー内で完了でき、登録操作から結果表示まで 1 秒以内（非機能要件.md の主要画面応答時間目標に準拠）で完了する。
- **SC-002**: 法人名＋住所が既存テナントと完全一致する重複法人による誤登録は、登録前の時点で 100% 検出され、二重テナントが作成されない（判定キーは Q-A1 の決定に基づく）。
- **SC-003**: ユーザー追加操作によって作成された新規ユーザーは、追加操作を行った既存ユーザーと同一の権限（同権限のみ、BR-003）で、追加操作完了直後からログイン可能になる。

## 入力（ユーザーが与える情報）

| 項目名 | 型 / 形式 | 必須 | 取りうる値 | 説明 |
|-------|---------|------|----------|------|
| 企業種別 | 単一選択 | 必須 | 配送依頼企業 / 運送会社 | どちらのテナント種別として登録するかを選択する |
| 法人名 | 文字列 | 必須 | — | テナントの識別名（重複判定キーの一部、Q-A1 決定済み） |
| 住所 | 文字列 | 必須 | — | 法人所在地（重複判定キーの一部、Q-A1 決定済み） |
| 会社電話番号 | 文字列 | 必須 | — | 代表電話番号 |
| 会社メールアドレス | 文字列（メール形式） | 必須 | — | 代表メールアドレス |
| 支払方法（種別） | 単一選択 | 必須 | 請求書払い（月末締め翌月末払い） / 銀行振込（Q-DM1 決定済み） | 支払方法の種別のみを登録項目とする |
| 利用規約・プライバシーポリシー同意 | チェック | 必須 | 同意 / 未同意 | 未同意の場合は登録できない |
| ユーザー名（識別名） | 文字列 | 必須 | — | 営業所名等の識別名も許容。案件詳細画面（SCR-009/SCR-015）の連絡履歴で送信者名として表示される（`交渉合意成約.md`、`画面一覧.md` 参照） |
| ログイン ID | 文字列 | 必須 | テナント内で一意（Q-NF1 決定済み） | ユーザー単位で個別発行。日常のログインはこの ID で行う |
| 登録メールアドレス | 文字列（メール形式） | 必須 | システム全体で一意（Q-NF1 決定済み） | ユーザー個人のメールアドレス。パスワードリセット（`認証.md` UC-006）の送信先として使用。ログイン ID とは独立した別項目 |
| ログイン用パスワード | 文字列 | 必須 | 最小 8 文字以上、英字（大文字・小文字問わず）と数字を各 1 文字以上含む（Q-NF4 決定済み） | — |

## 出力（システムが返す情報）

| 項目名 | 型 / 形式 | 説明 |
|-------|---------|------|
| テナント ID | 内部識別子 | 企業アカウントの一意識別子（設計フェーズで物理キーを確定） |
| ユーザー ID | 内部識別子 | 発行されたユーザーの一意識別子 |
| 登録結果メッセージ | 文字列（MSG-XXX） | 成功 / 重複エラー / 入力エラーのいずれか |

## 受け入れ条件

### 正常系

- [ ] AC-001: Given 企業共通項目・初回ユーザー項目をすべて有効な値で入力し利用規約に同意した状態 / When 登録操作を行う / Then テナントと初回ユーザーが作成され、ログイン画面へ遷移する
- [ ] AC-002: Given 既存テナントにログイン済みの状態 / When ユーザー追加操作で新規ユーザー項目を入力する / Then 同一テナント配下に新規ユーザーが追加され、同権限でログイン可能になる（UC-003）

### 異常系

- [ ] AC-101: Given 利用規約・プライバシーポリシーに同意していない状態 / When 登録操作を行う / Then MSG-018 が表示され登録は行われない
- [ ] AC-102: Given 必須項目が未入力、またはメールアドレス等の形式が不正な状態 / When 登録操作を行う / Then MSG-014 系の入力エラーが表示され登録は行われない

### 境界値・特殊条件

- [ ] AC-201: Given 入力された法人名＋住所が既存テナントの法人名＋住所と完全一致（正規化後）する状態（判定キーは Q-A1 決定済み） / When 登録操作を行う / Then 重複エラーが表示され新規テナントは作成されない

### 権限境界（該当する場合）

- 第 1 版はテナント内同権限のみ（BR-003）のため、テナント内でのロール別権限境界は存在しない。ただしテナントを跨いだ操作（他社テナントのユーザー追加等）はそもそも導線が存在せず、UI 上不可能とする。

### エッジケース

- AC-401: 法人重複と判定された場合の救済手段（既存テナントへの参加申請等）は第 1 版では提供しない（Q-A2 決定済み）。登録不可のみを表示し、既存テナント管理者への連絡を利用者に案内する（参加申請フローは将来拡張）。

## 前提条件（Assumptions）

- 企業種別（配送依頼企業 / 運送会社）は登録時に固定され、後から変更することはできない。
- ユーザー個別項目のうち、運送業許可番号・保有トラック詳細等は第 1 版では収集しない（要件素材 1-3 注記のとおり）。
- ログイン ID（テナント内一意）と登録メールアドレス（システム全体で一意）は独立した別フィールドであり、同一視しない（Q-NF1 決定済み）。
