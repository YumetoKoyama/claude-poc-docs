# シーケンス: SEQ-013 案件削除

## ID 凡例

| ID 体系 | 形式例 | 用途 |
|---------|-------|------|
| `SEQ-XXX` | `SEQ-013` | シーケンス ID |

## メタデータ

- シーケンス ID: SEQ-013
- シーケンス名: 案件削除（論理削除→夜間バッチ物理削除）
- 対応画面: SCR-009 案件詳細画面（配送依頼企業）, SCR-008 案件一覧画面
- 対応ユースケース: UC-009
- 対応業務フロー: ACT-005（案件削除フロー・論理削除→物理削除。`docs/requirements/activities/案件削除フロー.md`）
- 対応 API（operationId）: `deleteJob`
- 関連受け入れ条件: 案件削除/AC-001, 案件削除/AC-101, 案件削除/AC-201, 案件削除/AC-301, 案件削除/AC-401
- 関連業務ルール: BR-019, BR-022

## 受け入れ条件（Given/When/Then）

| AC-ID | 区分 | Given（前提状態） | When（API 呼び出し） | Then（期待結果） | 関連 BR |
|-------|------|-----------------|-------------------|----------------|--------|
| 案件削除/AC-001 | 正常系 | 案件ステータスが「募集中」または「交渉中」 | deleteJob | 204、論理削除（`deleted_at` 設定）＋応募していた運送会社へ通知（MSG-013） | BR-022 |
| 案件削除/AC-101 | 異常系 | 案件ステータスが「成約済」「運送中」「完了」「評価済」のいずれか | deleteJob | 導線非表示（UI）＋ API 側は 409 CONFLICT（MSG-021） | BR-019 |
| 案件削除/AC-201 | 境界値 | 論理削除されてから 24 時間未満 | getJobById / listJobs | 配送依頼企業側は「削除済み」表示を残す、運送会社側は非表示（Q-J8） | BR-022 |
| 案件削除/AC-301 | 権限境界 | 自社以外（他テナント）の案件 | deleteJob | 404 NOT_FOUND（テナント越境） | — |
| 案件削除/AC-401 | エッジケース | 論理削除から 24 時間経過 | （夜間バッチ EXT-010） | 物理削除実行、応募・連絡メッセージも同時物理削除、通知は保持（Q-DM4） | BR-022 |

## 前提条件

- 認証済み・配送依頼企業ユーザー
- 対象案件が自社（`requester_tenant_id` = JWT.tenantId）の登録案件

## シーケンス図（削除操作・同期処理）

```mermaid
sequenceDiagram
    autonumber
    actor User as 配送依頼企業ユーザー
    participant UI as フロントエンド
    participant API as バックエンド
    participant DB as データベース (jobs, bids, notifications)

    User->>UI: 削除操作
    UI->>User: 削除確認ダイアログ（MSG-012）
    User->>UI: 削除を確定
    UI->>API: DELETE /api/jobs/{jobId} (deleteJob, Authorization: Bearer)
    API->>API: 認可チェック（配送依頼企業ロールのみ）
    alt 運送会社ユーザー
        API-->>UI: 403 FORBIDDEN
    else 配送依頼企業ユーザー
        API->>DB: BEGIN TRANSACTION
        API->>DB: SELECT jobs WHERE id=? FOR UPDATE
        alt requester_tenant_id != JWT.tenantId
            API->>DB: ROLLBACK
            API-->>UI: 404 NOT_FOUND（テナント越境、案件削除/AC-301）
        else 自社案件
            alt status not in (RECRUITING, NEGOTIATING)
                API->>DB: ROLLBACK
                API-->>UI: 409 CONFLICT（案件削除/AC-101, MSG-021, BR-019）
            else
                API->>DB: UPDATE jobs SET deleted_at=now() WHERE id=?
                API->>DB: SELECT bids WHERE job_id=? AND status IN ('OPEN','CONTRACTED')
                DB-->>API: 応募していた運送会社一覧
                loop 応募していた運送会社ごと
                    API->>DB: INSERT notifications (recipient_tenant_id=carrier_tenant_id, related_job_id=jobId, type='GENERAL', body='MSG-013')
                end
                Note over API,DB: 通知作成に失敗しても削除操作自体は成功させる（通知・文面定義.md 2節）。通知INSERTの例外はログ記録のみでロールバック対象としない
                API->>DB: COMMIT
                API-->>UI: 204（案件削除/AC-001）
                UI-->>User: SCR-008（案件一覧）へ遷移、対象は「削除済み」表示
            end
        end
    end
```

## 削除済み表示（AC-201、参照系）

```mermaid
sequenceDiagram
    autonumber
    participant UI as フロントエンド
    participant API as バックエンド
    participant DB as データベース (jobs)

    Note over UI,DB: 配送依頼企業（自テナント）が一覧・詳細を参照する場合
    UI->>API: GET /api/jobs?scope=own (listJobs) / GET /api/jobs/{jobId} (getJobById)
    API->>DB: SELECT jobs WHERE requester_tenant_id=?（deleted_at 条件なし、自社は削除済みも表示）
    DB-->>API: 案件（deleted_at IS NOT NULL を含む）
    API-->>UI: 200、logicallyDeleted=true を含めて返却（案件削除/AC-201）

    Note over UI,DB: 運送会社（第三者テナント）が一覧・詳細を参照する場合
    UI->>API: GET /api/jobs (listJobs) / GET /api/jobs/{jobId} (getJobById)
    API->>DB: SELECT jobs WHERE deleted_at IS NULL（論理削除済みは除外）
    DB-->>API: 案件（削除済みを除く）
    API-->>UI: 200 / 404（削除済み案件への直接アクセスも404、案件削除/AC-201）
```

## 夜間バッチによる物理削除（AC-401、参照）

物理削除の詳細トランザクション設計（対象抽出条件・1件1トランザクション方針・再実行/冪等性・失敗時の扱い）は `バッチ設計.md` 2節「案件物理削除バッチ」（EXT-010）を正とする。本シーケンスでは同期処理（削除操作〜論理削除〜通知）との境界と、バッチ内の削除順序のみを示す（詳細は `バッチ設計.md` を参照）。

```mermaid
sequenceDiagram
    autonumber
    participant Batch as 案件物理削除バッチ（EXT-010, Spring @Scheduled）
    participant DB as データベース (jobs, bids, messages, job_attributes, notifications)

    Note over Batch: 深夜 2:00〜4:00（Q-NF9）に起動
    Batch->>DB: SELECT jobs WHERE deleted_at IS NOT NULL AND deleted_at <= now() - interval '24 hours'
    DB-->>Batch: 物理削除対象の案件一覧
    loop 対象案件ごと（1件1トランザクション、バッチ設計.md 2.3節）
        Batch->>DB: BEGIN TRANSACTION
        Batch->>DB: SELECT jobs WHERE id=? FOR UPDATE
        Batch->>DB: DELETE messages WHERE bid_id IN (SELECT id FROM bids WHERE job_id=?)（ENT-006、案件削除/AC-401）
        Batch->>DB: DELETE bids WHERE job_id=?（ENT-004、案件削除/AC-401）
        Batch->>DB: DELETE job_attributes WHERE job_id=?
        Batch->>DB: DELETE jobs WHERE id=?
        Batch->>DB: COMMIT
        Note over Batch,DB: notifications（ENT-009）は物理削除の対象としない。related_job_id は FK 制約なしのため孤立参照を許容し90日間保持（Q-DM4）。物理削除後に通知経由で該当案件へ遷移すると getJobById が404を返し FE は MSG-020 を表示する
    end
```

## 例外・代替フロー

| 例外区分 | 発生条件 | HTTP / エラーコード | 対応 AC / BR | 振る舞い |
|---------|---------|------------------|------------|---------|
| 認可失敗 | 運送会社ユーザーによる削除試行 | 403 FORBIDDEN | — | 削除導線自体を非表示（UI側）＋API側でも拒否 |
| テナント越境 | 他テナント案件への直接アクセス | 404 NOT_FOUND | 案件削除/AC-301 | 削除導線非表示、直打ちも404 |
| ステータスガード | 成約済以降の案件を削除 | 409 CONFLICT | 案件削除/AC-101 | MSG-021表示、削除導線自体を非活性（BR-019） |
| 通知作成失敗 | INSERT notifications 時の例外 | — | 案件削除/AC-001 | 削除操作自体は成功させる（204）。ログ記録のみで再送は行わない（通知・文面定義.md 2節） |
| 物理削除未到達 | 論理削除から24時間未満 | — | 案件削除/AC-201 | 配送依頼企業側は「削除済み」表示を残し、運送会社側からは非表示（Q-J8） |
