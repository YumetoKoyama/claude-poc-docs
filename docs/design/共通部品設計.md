# 共通部品設計（バックエンド）

> ID 凡例: [docs/凡例.md](../凡例.md) 参照
> 本ファイルは BE（Spring Boot）側の横断関心事のみを扱う。FE 側の横断関心事は `フロントエンド共通設計.md` を参照。
> ErrorResponse のスキーマ・エラーコード enum の正典は `api/_common.yaml`。本ファイルはその**実装方式**（例外クラス→HTTPステータス対応・マッピングロジック）のみを記述し、スキーマを再定義しない。

## 1. 共通例外ハンドラ

`@RestControllerAdvice` を `shared/` モジュールに配置し、全 Controller で共通の例外→`ErrorResponse` 変換を行う（`backend-04-package-structure.md` shared 構成例）。

### 1.1 例外クラス階層

`shared/exception/` に業務例外の基底クラスを定義し、各モジュールの業務例外はこれを継承する。

| 例外クラス | 継承元 | HTTP ステータス | 対応する `ErrorCode` enum（`_common.yaml` 参照、値は再掲しない） |
|-----------|--------|----------------|------------------------------------------------------|
| `ValidationException` | `DomainException` | 400 | `VALIDATION_ERROR` |
| `AuthenticationFailedException` | `DomainException` | 401 | `LOGIN_FAILED` |
| `AccountLockedException` | `DomainException` | 401 | `ACCOUNT_LOCKED` |
| `UnauthenticatedException` | `DomainException` | 401 | `UNAUTHENTICATED` |
| `ForbiddenException` | `DomainException` | 403 | `FORBIDDEN` |
| `NotFoundException` | `DomainException` | 404 | `NOT_FOUND` |
| `DuplicateException` | `DomainException` | 409 | `DUPLICATE` |
| `StateConflictException` | `DomainException` | 409 | `CONFLICT` |
| `BidLimitExceededException` | `DomainException` | 409 | `BID_LIMIT_EXCEEDED` |
| `BidDeadlinePassedException` | `DomainException` | 409 | `BID_DEADLINE_PASSED` |
| `SetBidTenantMismatchException` | `DomainException` | 400 | `SET_BID_TENANT_MISMATCH` |
| `ResetTokenExpiredException` | `DomainException` | 400 | `RESET_TOKEN_EXPIRED` |
| `RateLimitedException` | `DomainException` | 429 | `RATE_LIMITED` |
| （未捕捉の `Exception`） | — | 500 | `INTERNAL_ERROR` |

`DomainException` は抽象基底クラスとし、各サブクラスが自身に対応する `ErrorCode` enum 値（`_common.yaml` 参照）と HTTP ステータスを保持する。`@RestControllerAdvice` はサブクラスの型から自動的にステータス・コードを決定し、個別の `@ExceptionHandler` を業務例外の種類ごとに増殖させない（共通の 1 ハンドラで `DomainException` を受け、内部で保持情報から `ErrorResponse` を組み立てる）。

### 1.2 マッピングロジック

```
DomainException 発生
  → @RestControllerAdvice が catch
  → 例外クラスが保持する httpStatus / code（`_common.yaml` ErrorResponse.code の実フィールド名。ErrorCode enum 値）を取得
  → ErrorResponse{ code, message, details } を組み立てる
     - message: 例外に紐づく MSG-XXX 文言（メッセージ一覧.md、フロントエンド共通設計.md のマッピングと対応）
     - details: バリデーションエラー時のみ field/reason を設定（ValidationException のみ）
  → ResponseEntity<ErrorResponse> をステータスコード付きで返却
```

- `MethodArgumentNotValidException`（Bean Validation 由来）は専用ハンドラで捕捉し、`VALIDATION_ERROR` + `details[]`（field/reason）に変換する。
- 未捕捉の `Exception` は `INTERNAL_ERROR` として 500 を返し、スタックトレース等の内部情報はレスポンスに含めずログにのみ記録する（`backend-07-security-coding.md` R-SEC-040）。

## 2. バリデーション共通化

- フォーマット系バリデーション（必須・桁数・パターン等）は OpenAPI YAML のベンダー拡張（`x-validate-not-blank` 等）から生成される Bean Validation アノテーションで行う（`backend-00-stack.md` #16、`backend-02-layers.md`）。
- 相対条件・相関制約（例: `from_datetime < to_datetime`、応募締切の「from時刻の2時間前」判定）は生成クラスに表現できないため、UseCase 内の `Validator`（`shared/` 配下の共通インターフェースを各モジュールで実装）で行う（`backend-07-security-coding.md` §0(b)）。
- 業務ルールバリデーション（ステータス遷移可否、BR-019 の編集制限等）は Controller・共通 Validator ではなく domain 層のメソッド内でガードする（`backend-07-security-coding.md` R-SEC-021）。

## 3. 共通レスポンス整形

- 成功レスポンスは各 operationId の Response DTO（`generated.openapi.*`）をそのまま返す。共通のラッパー（`{data: ...}` 等）は導入しない（`api/*.yaml` のレスポンススキーマがそのまま JSON ボディ）。
- ページング応答は `_common.yaml` の `PageMeta` を用いる（`page`/`size`/`totalElements`/`totalPages`）。一覧系 operationId（`listJobs`, `listMyBids`, `listNotifications` 等）はレスポンスに `PageMeta` を含める。
- エラーレスポンスは本ファイル 1 節のとおり `ErrorResponse`（`_common.yaml` 正典）に統一する。

## 4. ロギング方式

| 項目 | 方針 |
|------|------|
| ログフレームワーク | Spring Boot 標準（Logback） |
| ログレベル | INFO: 業務操作の開始/終了（成約・削除等の重要イベント）。WARN: バリデーションエラー・404/409等の業務例外。ERROR: 未捕捉例外・外部IF（メール送信・バッチ）失敗 |
| 機微情報のマスク | パスワード・トークン・個人情報（氏名・電話番号・メールアドレス）をログに出力しない（R-SEC-050）。Request/Response の機微フィールドはログ出力対象から除外する（`_common.yaml` の `x-sensitive: true` フィールドと対応） |
| トレース | 例外ログにはリクエスト ID（MDC）を含め、障害調査での追跡を可能にする（R-SEC-052） |
| 出力先 | 標準出力（コンテナ運用を前提に、集約先はインフラ構築フェーズで確定。`運用設計.md` 参照） |

## 5. 共通例外ハンドラ以外の横断部品

| 部品 | 役割 | 配置 |
|------|------|------|
| `Clock`（interface） | 現在時刻取得の抽象化。`LocalDateTime.now()` の直接呼び出しを禁止（R-ARCH-010） | `shared/clock/Clock.java`（interface）、`infrastructure` 側に `SystemClock` 実装 |
| 共通識別子（`TenantId`, `UserId`, `JobId` 等） | 型安全な ID 表現、Mass Assignment 防止の一助 | `shared/id/` |
| `MailSender`（Port） | メール送信の抽象化（EXT-001） | `auth/infrastructure/` に実装（`backend-01-architecture.md` R-ARCH-007） |
| `NotificationPublisher`（Port） | 通知発火の抽象化（Domain Event 経由、R-ARCH-008） | `notification/infrastructure/` |
| `EnumMapper` | ドメイン enum ↔ OpenAPI 生成 inline enum の変換 | 各モジュールの Controller 境界（`backend-02-layers.md`） |

## 6. 通知宛先粒度

通知（`notifications`）の宛先粒度は**テナント単位**とする（`概要.md` 前提条件、`tables/notifications.md` 参照）。`NotificationPublisher` はテナント ID を宛先として通知レコードを作成する。ユーザー単位への分離は将来課題とする。
