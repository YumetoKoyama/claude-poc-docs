# IF 定義（外部・内部インターフェース）

> ID 凡例: [docs/凡例.md](../凡例.md) 参照

要件 `外部インターフェース一覧.md` の EXT-001（パスワードリセットメール送信）・EXT-010（案件物理削除バッチ）・EXT-011（通知物理削除バッチ）を設計レベルへ物理化する。第 1 版では上記 3 件以外の外部 IF は存在しない（外部システム連携なし、`外部インターフェース一覧.md` 4節）。

## 1. 内部 IF（FE ↔ BE）

FE ↔ BE 間のインターフェースは OpenAPI 3.1（`docs/design/api/*.yaml`）を正典とする。本書では再定義しない。全 API は `Authorization: Bearer <accessToken>` ヘッダによる認証を前提とする（`login`/`registerTenant`/`requestPasswordReset`/`resetPassword`/`refreshAccessToken` を除く。`セキュリティ設計.md` 参照）。

## 2. EXT-001: パスワードリセットメール送信

| 項目 | 内容 |
|------|------|
| IF 種別 | メール送信（送信のみ、応答なし） |
| 起動契機 | `requestPasswordReset`（POST /api/auth/password-reset-requests）のレスポンス処理内、同期実行 |
| 送信元コンポーネント | BE（Spring Boot）の通知送信コンポーネント（`共通部品設計.md` 参照） |
| 送信先 | ユーザーの登録メールアドレス（`users.email`） |
| プロトコル | SMTP（送信方式の実装詳細はデプロイ環境依存。方式設計.md 参照） |
| テンプレート | パスワードリセット案内メール（`通知・文面定義.md` 1節）。再設定 URL・有効期限（発行から1時間、Q-NF3）を含む |
| トランザクション境界 | 業務トランザクション（リセットトークン発行・`password_reset_tokens` への INSERT）とメール送信は分離する。メール送信失敗時も API レスポンスは 202 のまま変えない（`通知・文面定義.md` 2節、ユーザー列挙防止） |
| 失敗時挙動 | 送信失敗はアプリケーションログに記録し、業務操作自体（トークン発行）は成功のまま維持する。継続的な送信失敗はアプリケーションの死活監視（`運用設計.md`）で検知し、運用担当者へメール通知する（Q-NF8） |
| レート制限 | 同一メールアドレスに対し 1 時間あたり 5 回まで（`auth.yaml` `requestPasswordReset` 参照、429 TooManyRequests） |

## 3. EXT-010: 案件物理削除バッチ

詳細トランザクション設計・再実行方針・監視連携は `バッチ設計.md` を正典とする。本節では IF としての入出力インターフェースのみを定義する。

| 項目 | 内容 |
|------|------|
| IF 種別 | バッチ（内部、外部システム連携なし） |
| 起動契機 | cron（毎日深夜 2:00〜4:00 の間、Q-NF9） |
| 入力 | `jobs.deleted_at IS NOT NULL AND deleted_at <= now() - 24h` を満たす案件 |
| 出力 | 対象案件（`jobs`）・紐づく応募（`bids`）・連絡メッセージ（`messages`）の物理削除。通知（`notifications`）は対象外（保持、Q-DM4） |
| 冪等性 | 冪等（同一対象への再実行は 0 件処理となり副作用なし） |
| 失敗時挙動 | 当日実行分をスキップし翌日の実行で再試行。継続失敗はアプリケーションの死活監視で検知し運用担当者へメール通知（Q-NF8） |

## 4. EXT-011: 通知物理削除バッチ

詳細トランザクション設計・再実行方針・監視連携は `バッチ設計.md` 3節を正典とする。本節では IF としての入出力インターフェースのみを定義する。

| 項目 | 内容 |
|------|------|
| IF 種別 | バッチ（内部、外部システム連携なし） |
| 起動契機 | cron（毎日深夜 2:00〜4:00 の間、EXT-010 とは別ジョブ、Q-NF9 準用） |
| 入力 | `notifications.created_at <= now() - 90日` を満たす通知 |
| 出力 | 対象通知（`notifications`）の物理削除（保管期間 90 日、Q-NF10） |
| 冪等性 | 冪等（同一対象への再実行は 0 件処理となり副作用なし） |
| 失敗時挙動 | 当日実行分をスキップし翌日の実行で再試行。継続失敗はアプリケーションの死活監視で検知し運用担当者へメール通知（Q-NF8） |

## 5. 対象外として明示する外部 IF

要件 `外部インターフェース一覧.md` 5節（決済代行 API、地図/ジオコーディング API、リアルタイム Push 通知基盤、SSO/外部 ID プロバイダ、外部 CRM 連携）を設計対象外として踏襲する。第 1 版ではいずれも実装しない。
