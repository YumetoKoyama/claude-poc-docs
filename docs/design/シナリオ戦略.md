# E2Eシナリオ方針

> ID 凡例: [docs/凡例.md](../凡例.md) 参照
> **凍結中の明示**: E2E の実行は AWS 環境構築後に `claude-poc-e2e` リポジトリの別工程で行う（`/e2e-from-design` は凍結中、CLAUDE.md 開発フロー 3節）。本書は将来の E2E 実施に備えたシナリオ方針の定義のみを行い、本設計フェーズ・製造フェーズでは E2E の作成・実行を行わない。

## E2E カバー範囲

| 対象フロー | 優先度 | 説明 |
|----------|-------|------|
| 案件成約フロー（ACT-001） | 高 | 案件登録 → 応募 → 交渉 → 最終条件提示 → 合意 → 成約までの一連 |
| セット応募フロー | 高 | セット応募 → 一括合意 → セット内全案件成約、または一部不成立時の連動クローズ |
| 運送実施フロー（ACT-002） | 高 | 運送開始報告 → 完了報告 → 完了確認 |
| 評価フロー（ACT-003） | 中 | 双方評価登録 → ステータス「評価済」への遷移 |
| アカウント登録・認証フロー | 高 | 企業アカウント登録 → ログイン → ログアウト、パスワードリセット |
| 案件削除フロー（ACT-005） | 中 | 案件削除（論理削除）→ 通知確認 → （バッチ実行は対象外、単体/結合で検証） |
| 通知フロー | 低 | 通知アイコン確認 → 通知一覧 → 該当案件への遷移・既読化 |
| 取引履歴閲覧 | 低 | 成約済〜評価済案件の履歴閲覧 |

| E2E-001 | 案件登録から成約までの一気通貫 | SCR-007, SCR-008, SCR-009, SCR-014, SCR-015 | 配送依頼企業・運送会社の各テナントがログイン可能な状態 | 案件登録→応募→最終条件提示→合意 | 案件ステータスが「成約済」に遷移し成約スナップショットが保存される（AC-001 交渉合意成約.md） |
| E2E-002 | セット応募の一括合意 | SCR-014, SCR-015-01, SCR-009-01 | 同一配送依頼企業の複数募集中案件が存在する状態 | セット応募→セット一括合意 | セット内すべての案件が一括で成約済みになる（AC-002 交渉合意成約.md） |
| E2E-003 | セット応募の連動不成立 | SCR-014, SCR-015 | セット応募済みの案件の一部が他社に先に成約される状態を再現 | 一部案件の他社成約を発生させる | セット応募全体が自動不成立になり通知（MSG-007）が届く（AC-201 交渉合意成約.md） |
| E2E-004 | 運送実施から完了確認まで | SCR-009, SCR-015 | 案件が成約済の状態 | 運送開始報告→完了報告→完了確認 | 案件ステータスが「運送中」→「完了」に遷移する（運送ステータス報告.md AC-001, AC-002） |
| E2E-005 | 双方評価登録 | SCR-010, SCR-017 | 案件が完了ステータスの状態 | 双方が評価登録 | 双方評価完了で案件ステータスが「評価済」に遷移する（評価.md AC-001） |
| E2E-006 | アカウント登録とログイン | SCR-002, SCR-001 | 未登録の企業情報 | 企業アカウント登録→ログイン | 登録直後にログイン画面へ遷移し、発行IDでログインできる（アカウント登録.md AC-001） |
| E2E-007 | パスワードリセット | SCR-003, SCR-004, SCR-001 | 登録済みメールアドレスを保有 | リセット申請→メールリンク→新パスワード設定→ログイン | 新パスワードでログインできる（認証.md SC-003） |
| E2E-008 | 案件削除と通知 | SCR-009, SCR-012 | 応募済みの募集中案件が存在 | 案件削除操作 | 応募していた運送会社へ通知が届く（案件削除.md AC-001） |
| E2E-009 | 通知経由の案件遷移 | SCR-012, SCR-009, SCR-015 | 未読通知が存在 | 通知一覧から通知クリック | 該当案件詳細へ遷移し既読化される（通知.md AC-002） |
| E2E-010 | 取引履歴閲覧 | SCR-011, SCR-018 | 成約済〜評価済の過去案件が存在 | 取引履歴一覧を開く | 自社が当事者の案件のみ表示される（取引履歴.md AC-001） |

## カバー観点

- 正常系（ゴールデンパス）: E2E-001, E2E-002, E2E-004, E2E-005, E2E-006 の一連のハッピーパス。
- 入力エラー系: 単体/結合テストで担保し、E2E では代表的な 1 パターン（例: 案件登録の必須未入力）のみを対象とする（網羅は単体/結合の責務、E2E はコスト最適化のため主要フローに限定）。
- 状態遷移: 案件ステータス（募集中→交渉中→成約済→運送中→完了→評価済）・応募ステータス（応募中→成約/クローズ）を E2E-001〜E2E-005 で一気通貫検証する。
- 永続化確認: 成約スナップショット（ENT-007）・評価（ENT-008）・通知（ENT-009）の永続化結果を画面表示で確認する（DB直接確認ではなく画面経由の確認とする）。

## 実行環境・テストデータ方針

- 実行環境: AWS 環境構築後、`claude-poc-e2e` リポジトリで Playwright MCP 前提の E2E を実行する（現時点では未構築、`/e2e-from-design` 凍結中）。
- テストデータ準備方法: 配送依頼企業・運送会社それぞれのテストテナント・テストユーザーを事前投入する。マスタデータ（支払方法・物品種別等）は `移行要件.md` MIG-001/MIG-002 の初期データを使用する。
- データクリーンアップ方針: テストシナリオごとに独立したテストデータ（案件・応募）を新規作成し、既存データとの干渉を避ける。E2E 実行環境のデータは定期的にリセットする運用を想定する（詳細は E2E 環境構築フェーズで確定）。
