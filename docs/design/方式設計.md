## 1. 論理アーキテクチャ

FE（React/Next.js）と BE（Spring Boot）を別リポジトリ・別デプロイ単位とし、OpenAPI 3.1（`docs/design/api/`）で定義した REST API のみで結合する。BE は単一 Spring Boot アプリケーションのモジュラモノリス（`claude-poc-backend/.claude/rules/backend-01-architecture.md` 準拠）とし、マイクロサービス分割は行わない。

## 3. 環境別デプロイ構成

第 1 版はデモ用途（`非機能要件.md` 可用性節）であり、目標稼働率・RTO/RPO は明示しない。本番方式としては将来の AWS 環境移行を想定するが、具体的なクラウドインフラ構成（VPC・マネージドサービス選定等）は本設計時点で未確定であり、インフラ構築フェーズ（E2E リポジトリ・AWS 環境構築、`CLAUDE.md` 記載）で別途確定する。本書では環境間の差分方針のみを定義する。

| 環境 | FE | BE | DB | 用途 |
|------|----|----|----|----|
| ローカル開発 | `npm run dev`（Next.js dev server） | `mvn spring-boot:run` | devcontainer 内 PostgreSQL 16 コンテナ | 個人開発・動作確認 |
| CI | ビルド・単体テスト実行のみ（常駐なし） | ビルド・単体テスト・結合テスト（Testcontainers PostgreSQL 16）実行のみ | Testcontainers（テスト実行時のみ起動） | Pull Request 検証 |
| 本番相当（将来） | コンテナ化した Next.js（Node ランタイム） | コンテナ化した Spring Boot アプリケーション | マネージド PostgreSQL 16（想定） | デモ公開環境（インフラ詳細は AWS 環境構築フェーズで確定） |

- 環境変数（DB 接続情報・JWT シークレット・CORS 許可オリジン・SMTP 接続情報等）は環境ごとに外部設定で注入し、ソースコードにハードコードしない（`backend-07-security-coding.md` R-SEC-032）。
- FE と BE は別デプロイ単位とし、互いのバージョンを独立してリリースできるようにする（結合点は OpenAPI 契約のみ）。

## 4. ミドルウェア構成

| ミドルウェア | 役割 | 備考 |
|------------|------|------|
| PostgreSQL 16 | 業務データの永続化 | `DB定義.md` 参照 |
| SMTP（メール送信） | パスワードリセットメール送信（EXT-001） | 具体的な送信サービス（SES 等）はインフラ構築フェーズで確定。BE はメール送信を Port/Adapter（`MailSender`）で抽象化し実装差し替えを容易にする（`backend-01-architecture.md` R-ARCH-007） |
| リバースプロキシ / ロードバランサ | HTTPS 終端、FE/BE へのルーティング | 具体構成はインフラ構築フェーズで確定。HTTPS 必須（`非機能要件.md`） |

## 5. 通信方式

- FE ↔ BE: HTTPS 上の REST API（JSON）。認証は JWT（Bearer トークン）。CORS 許可オリジンは `セキュリティ設計.md` で確定値を管理する。
- BE ↔ DB: JDBC（Spring Data JPA 経由）。
- BE → SMTP: メール送信 IF（EXT-001、`IF定義.md` 参照）。

## 6. スケーラビリティ・パフォーマンス方針

- 想定同時アクセス数は最大 100 ユーザー（`非機能要件.md`）であり、第 1 版では水平スケーリング・オートスケール構成は必須としない。単一インスタンス構成を前提に設計する。
- 応募枠・成約処理等の高競合操作は悲観ロック（`tables/jobs.md` 排他制御節）で整合性を保証するため、複数インスタンス運用時はロック範囲が単一 DB トランザクションに閉じることを前提とする（アプリケーションサーバーを複数台に増やしても DB 側のロックにより整合性は維持される）。
