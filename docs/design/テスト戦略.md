# 単体テスト方針

> ID 凡例: [docs/凡例.md](../凡例.md) 参照
> テスト 3 層（単体/結合/E2E）・区分（正常系/異常系/境界値/権限境界）・RTM 列構成の正典は shared-canon §2（親 CLAUDE.md）。本書は単体テスト（TC-XXX）の方針のみを定義する。単体マトリクス・RTM の実成果物は製造フェーズ（`/test-design-from-issue`）で `docs/test/` に作成する。

## テスト対象範囲

| レイヤー | 対象クラス | テスト方針 |
|--------|----------|-----------|
| presentation（Controller） | `*CommandController`, `*QueryController` | `MockMvc` を使用（`backend-06-coding-standards.md` R-TEST-001。`@Valid @RequestBody` を持つメソッドは必ず MockMvc 経由で Bean Validation を実行させる） |
| application（UseCase） | `Create/Update/Delete*UseCase` 等 | Mockito でリポジトリ・Port をモックし、業務ルールの分岐（正常/異常/境界値）を単体検証 |
| domain（Entity/Value Object/Domain Service） | `Job`, `Bid`, `SetBid`, `Deal`, `BidLimitPolicy` 等 | モック不要の純粋単体テスト。状態遷移・不変条件（BR-XXX）を網羅 |
| infrastructure（Repository実装） | `*RepositoryImpl` | `@DataJpaTest` + Testcontainers（結合寄りだが単体テスト層に含める。テナントフィルタ・一意制約の検証を含む） |
| FE（コンポーネント/hooks/store） | 各 `*.tsx` / `*.store.ts` | Vitest + React Testing Library（`frontend-testing-naming.md`）。ユーザー操作は `userEvent` を使用（`fireEvent` 原則禁止） |
| FE（バリデーション/フォーマッタ） | Zod スキーマ・ユーティリティ | 純粋関数の単体テスト |

## テスト種別と優先度

| 優先度 | 種別 | 内容 |
|-------|------|------|
| 高 | 業務ルール | BR-004〜BR-025 の状態遷移・制約（応募上限20社・自動締切・セット応募オール・オア・ナッシング・成約スナップショット・編集制限等）を domain 層で網羅 |
| 高 | 入力検証 | 各 `api/*.yaml` の required/minLength/maxLength/pattern/enum に対応する境界値（許容最小/最大、境界外） |
| 高 | 権限境界 | テナント越境（404）・自テナント内アクセス可否（`セキュリティ設計.md` 5.3 の全 operationId） |
| 中 | 分岐 | ステータス別の操作可否分岐（募集中/交渉中/成約済/運送中/完了/評価済） |
| 中 | 例外処理 | `共通部品設計.md` 1節の例外クラス→`ErrorResponse`変換の網羅 |
| 低 | 参照系（query） | 一覧・集計の表示整形（P2 領域、`backend-05-domain-priority.md`） |

## テスト境界とモック方針

- 外部依存（DB・外部 IF）のモック方針: UseCase 単体テストでは Repository/Port（`MailSender`, `NotificationPublisher`, `Clock`）を Mockito でモックする。`Clock` は固定時刻を返すモックとし、応募締切（BR-010）・トークン有効期限（Q-NF3）等の時刻依存ロジックを決定的にテストする。
- テストデータ管理方針: FE は `src/test/fixtures/` に fixture を配置し MSW（Mock Service Worker）で API をモックする（`frontend-testing-naming.md`）。BE は各テストクラス内でビルダー/ファクトリメソッドによりテストデータを組み立てる。
- カバレッジ目標: BE は JaCoCo INSTRUCTION 100%・BRANCH 90%（除外後、`backend-00-stack.md` #13）。FE は Vitest カバレッジ 100%（除外後、`frontend-00-stack.md` #10）。未達時は `coverage-to-100` skill で補完する。

## 要件とテストの対応方針

- 要件定義の受け入れ条件（AC-XXX、各画面 md「受け入れ条件」節・各 `functional/*.md`）ごとに、少なくとも 1 つの実行可能テスト（TC-XXX）を対応付ける。
- 各 operationId の `x-ac`（`api/*.yaml`）と AC-XXX の対応を起点に、TC-XXX を Controller/UseCase/Domain の各層へマッピングする。
- 対応表（RTM: UC / AC / BR / SCR / operationId / Issue# / TC-XXX / IT-XXX / E2E-XXX）は `docs/test/RTM.md` および単体テストマトリクスは `docs/test/単体テストマトリクス.md` で管理する（製造フェーズ、`/test-design-from-issue` が生成）。
