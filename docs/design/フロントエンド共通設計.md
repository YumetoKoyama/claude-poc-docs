# フロントエンド共通設計

> ID 凡例: [docs/凡例.md](../凡例.md) 参照
> 本ファイルは FE（React/Next.js）側の横断関心事のみを扱う。BE 側の横断関心事は `共通部品設計.md` を参照。
> 技術スタック（バージョン等）は `claude-poc-frontend/.claude/rules/frontend-00-stack.md` を正典とし再掲しない。ディレクトリ構成・コンポーネント分類・状態管理・ルーティングの実装規約詳細は同リポジトリの `frontend-*.md` 群を正典とし、本ファイルは業務横断の対応関係（エラー表示・API境界・認可ガード方針）のみを定義する。

## 1. 共通 UI コンポーネント方針

Atomic Design 5 階層（`frontend-component-design.md`）に従う。本アプリで共通化すべき業務非依存コンポーネント（atoms/molecules）と、複数画面で再利用される業務コンポーネント（organisms）を以下に列挙する。

| 分類 | コンポーネント | 用途 |
|------|--------------|------|
| atoms | Button, Input, Select, Checkbox, StarRating（★評価入力/表示） | 全画面共通の基礎部品 |
| atoms | Badge（ステータスバッジ: 募集中/交渉中/成約済/運送中/完了/評価済） | `JobStatus`（`_common.yaml`）の表示。表示名は `x-display-names` と一致させる |
| molecules | FormField（ラベル+入力+バリデーションエラー表示） | 全入力フォーム共通 |
| molecules | Toast（グローバルエラー/成功通知） | `共通レイアウト.md` の「グローバルエラー表示」 |
| molecules | ConfirmDialog（確認ダイアログ） | 削除確認（MSG-012）等 |
| organisms | AppHeader, GlobalNav, Footer | `screens/共通レイアウト.md` の共通シェルを構成 |
| organisms | NotificationBell, NotificationList | 通知アイコン・通知一覧（SCR-012） |
| organisms | JobCard / BidCard | 案件一覧・応募一覧の表示単位（配送依頼企業/運送会社で共用） |
| organisms | MessageThread（連絡履歴） | SCR-009/SCR-015 の交渉画面で共用 |

## 2. API クライアント

- 共通 HTTP クライアントは `src/lib/api/client.ts` に配置する（`frontend-state-api.md`）。
- 責務: `Authorization: Bearer <accessToken>` ヘッダ付与、401 発生時の `refreshAccessToken` 自動実行とリトライ、CORS 関連ヘッダの付与不要（Cookie 非使用）。
- `src/lib/api/errors.ts` に `ErrorResponse`（`_common.yaml` 正典）を受け取り、後述 3 節のマッピングに従って表示区分（フィールド直下 / トースト / リダイレクト）を判定する共通関数を置く。
- 機能別 API 呼び出しは `src/features/*/services/*.ts` に配置し、エンドポイント URL・operationId 単位の型（Request/Response）を明示する（画面コンポーネントで URL を直書きしない）。
- レスポンス DTO 型は `docs/design/api/*.yaml` の schema と一致させる（フィールド名を一字一句一致させる、RC-06）。

## 3. ErrorResponse.code → 表示文言（MSG-XXX）マッピング

`_common.yaml` の `ErrorCode` を正典とし、以下のマッピングで表示文言・表示区分を決定する。文言は要件 `メッセージ一覧.md` の MSG-XXX を正とし、本ファイルでは複製せず引用する。

| ErrorCode | 表示区分 | 表示文言 | 備考 |
|-----------|---------|---------|------|
| `VALIDATION_ERROR` | フィールド直下（`details[].field`ごと） | `ErrorResponse.message` をそのまま表示（サーバー側で MSG-001/MSG-002/MSG-008/MSG-014/MSG-018 等コンテキストに応じた文言を設定済み） | `共通部品設計.md` 1節のマッピングロジックで BE 側が文言を確定する |
| `UNAUTHENTICATED` | リダイレクト（トースト表示なし） | — | `refreshAccessToken` 失敗時は SCR-001 へ強制遷移（`共通レイアウト.md` 参照） |
| `LOGIN_FAILED` | フィールド直下（SCR-001） | MSG-015 | |
| `ACCOUNT_LOCKED` | フィールド直下（SCR-001） | MSG-016 | |
| `FORBIDDEN` | トースト | MSG-023 | テナント越境等は通常 `NOT_FOUND` になるため本コードは自テナント内の異常系のみ |
| `NOT_FOUND` | 文脈依存（トースト or 専用表示） | 通知遷移時は MSG-020。それ以外は MSG-025 | テナント越境の隠蔽結果としての404を含む |
| `DUPLICATE` | フィールド直下 or トースト | 応募重複は MSG-017。法人重複（登録画面）は MSG-024（AC-201） | |
| `CONFLICT` | トースト | 文脈により MSG-009 / MSG-010 / MSG-011 / MSG-021 / MSG-022 のいずれか（サーバー `message` に確定文言を含む） | 状態競合・楽観ロック競合共通 |
| `BID_LIMIT_EXCEEDED` | トースト | MSG-004 | |
| `BID_DEADLINE_PASSED` | トースト | MSG-005 | |
| `SET_BID_TENANT_MISMATCH` | フィールド直下（セット対象選択 UI） | MSG-003 | |
| `RESET_TOKEN_EXPIRED` | 専用表示（SCR-004） | MSG-019 | |
| `RATE_LIMITED` | トースト | MSG-026 | パスワードリセット申請の429 |
| `INTERNAL_ERROR` | トースト | 「内部エラーが発生しました。時間をおいて再度お試しください。」（MSG 未割当の汎用文言） | スタックトレース等は表示しない |

> `INTERNAL_ERROR` のみ要件 `メッセージ一覧.md` に対応行が無いため、実装時の暫定文言として本ファイルに記載する（500エラーは詳細文言を作り込まずスタックトレース非表示のみを担保する方針のため、MSG 番号化の優先度は低いと判断）。

## 4. 共通バリデーション

- Zod（`frontend-testing-naming.md` 採用ツール）でスキーマを定義し、`src/features/*/schemas/*.ts` に配置する。
- BE の Bean Validation 制約（`minLength`/`maxLength`/`pattern`/`required`）と値を一致させる（`api/*.yaml` を正典として Zod スキーマを手動同期する。将来的な自動生成は本設計のスコープ外）。
- 相対条件（例: from時刻 < to時刻）は Zod の `refine` で表現し、エラーメッセージは対応する MSG-XXX を使用する。

## 5. ルーティング・認可ガード方針

`frontend-routing.md` の 3 層構成をそのまま採用する。

1. **middleware.ts**: 未ログインでの `(protected)` 配下アクセスを `/login`（SCR-001）へリダイレクト。`tenantType` に基づき `/carrier/*` ↔ `/shipper/*` の相互アクセスを `/forbidden` へリダイレクト。
2. **`(protected)/layout.tsx`**: `getCurrentUser`（画面リロード時の D-2 対応、`共通レイアウト.md` 参照）で認証情報を取得し `AppHeader`/`GlobalNav` に渡す。取得失敗時は `/login` へ戻す。
3. **page.tsx / feature 側**: ボタン活性・編集可否等の業務単位の表示制御（例: BR-019 のステータスガードによる編集/削除ボタン非表示）。権限判定ロジックは `common/permissions` に集約する。

`REQUESTER`/`CARRIER` のロール別画面は要件 `画面一覧.md` の SCR-XXX に対応する URL セグメント（`shipper`/`carrier`）に分離する（`frontend-directory-structure.md`）。

## 6. 状態管理方針

- Zustand を UI state（モーダル開閉・フィルタ条件・選択状態・認証コンテキストの一部）に限定して使用する（`frontend-state-api.md`）。
- サーバーデータ（案件一覧・応募一覧等）は Zustand へ永続キャッシュせず、画面表示のたびに API から取得する（第 1 版はキャッシュ層を導入しない確定方針、`frontend-00-stack.md` #7）。
- 認証トークン（アクセストークン・リフレッシュトークン）は `localStorage`/`sessionStorage` 等の機密情報保存禁止対象に該当するため保存しない（`frontend-coding-rules.md` 3節）。アクセストークンはメモリ内（Zustand の認証 store、非永続）に保持し、リフレッシュはページロード時に `refreshAccessToken` を都度実行して復元する方式とする。

## 7. 共通レイアウトとの関係

`screens/共通レイアウト.md` の共通シェル（ヘッダー・グローバルナビ・フッター）は本ファイルの `AppHeader`/`GlobalNav`/`Footer` organisms として実装する。データ源（operationId.フィールド）は `共通レイアウト.md` の表を正とする。
