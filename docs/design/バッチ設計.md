# バッチ設計

> ID 凡例: [docs/凡例.md](../凡例.md) 参照

要件 `外部インターフェース一覧.md` 3節（EXT-010・EXT-011）に対応する、第 1 版の 2 つのバッチ「案件物理削除バッチ」「通知物理削除バッチ」を設計する。第 1 版ではこれ以外のバッチは存在しない。

## 1. バッチ一覧

| バッチ名 | EXT ID | 実行方式 | 実行時間枠 | 対応モジュール |
|---------|--------|---------|-----------|--------------|
| 案件物理削除バッチ | EXT-010 | Spring `@Scheduled`（cron） | 深夜 2:00〜4:00（Q-NF9） | `job/`（`backend-03-modules.md`）。専用バッチ実行基盤は第 1 版では導入せず BE アプリケーションに内蔵する |
| 通知物理削除バッチ | EXT-011 | Spring `@Scheduled`（cron） | 深夜 2:00〜4:00（EXT-010 と別ジョブ、Q-NF9 準用） | `notification/`（`backend-03-modules.md`）。専用バッチ実行基盤は第 1 版では導入せず BE アプリケーションに内蔵する |

## 2. 案件物理削除バッチ 詳細設計

### 2.1 目的・トリガ

`jobs.deleted_at IS NOT NULL AND deleted_at <= now() - 24h` を満たす案件（論理削除から24時間経過、BR-022）を対象に、案件・紐づく応募・連絡メッセージを物理削除する。

### 2.2 実行スケジュール

- cron 式で深夜 2:00〜4:00 の間に 1 回実行する（実行開始時刻は環境変数で設定可能とし、具体時刻はインフラ構築フェーズで確定。Q-NF9 の枠内であればよい）。
- 実行時間枠を 2 時間確保しているのは、想定データ規模（最大300件、`非機能要件.md`）に対して十分な余裕を持たせるため。

### 2.3 トランザクション単位

- **対象案件 1 件を 1 トランザクション**とする（全件一括の単一巨大トランザクションにしない）。1 件ごとに以下を同一トランザクション内で処理する。
  1. `jobs.id = ?` の対象案件を悲観ロック（`SELECT ... FOR UPDATE`）で取得（他の同時実行・手動操作との競合防止）。
  2. `messages`（当該案件配下の全応募に紐づくメッセージ）を削除。
  3. `bids`（当該案件の全応募）を削除。
  4. `job_attributes`（当該案件の属性）を削除。
  5. `jobs`（当該案件本体）を削除。
  6. `notifications` は削除しない（Q-DM4、保持期間90日は独立して `related_job_id` を参照可能なまま保持する。関連案件削除後の参照は `getJobById` が404を返しFEがMSG-020を表示、`tables/notifications.md` 参照）。
- 1 件単位でトランザクションを分けることで、1 件の失敗が他の対象案件の処理をブロックしない（部分的な成功を許容する）。

### 2.4 再実行・冪等性

- 冪等: 削除条件（`deleted_at <= now() - 24h`）を満たさなくなった対象（＝既に削除済み）は再実行時に抽出されないため、同一対象への重複実行は発生しない。
- 失敗した案件は当日の実行対象から除外されず、翌日以降の実行で再度抽出・再試行される（削除条件を満たし続ける限り）。

### 2.5 分割方針

- 対象件数が少量（想定最大300件規模、`非機能要件.md`）であるため、バッチ内チャンク分割（ページング処理）は必須としない。1 実行で全対象を順次処理する。
- 将来データ規模が拡大した場合は、`idx_jobs_updated_at`（`tables/jobs.md`）等を活用したページング処理への拡張を検討する。

### 2.6 監視連携

- 実行結果（成功件数・失敗件数・実行時間）をアプリケーションログに記録する。
- アプリケーションの死活監視（ヘルスチェック、`運用設計.md`）とは別に、バッチの継続失敗（複数日にわたる失敗）を検知した場合は運用担当者へメール通知する（Q-NF8）。監視方式の実装詳細（ログ集計・アラート発火条件）は `運用設計.md` を参照。

### 2.7 失敗時詳細

| 失敗パターン | 挙動 |
|------------|------|
| 特定 1 件の削除処理中の例外（DB制約違反等） | 当該案件のトランザクションのみロールバック。ログにエラー記録。他の対象案件の処理は継続する |
| バッチ全体の起動失敗（DB接続不可等） | 当日の実行をスキップし、翌日の実行枠で再試行する |
| 継続的な失敗（複数日） | 死活監視で検知し運用担当者へメール通知（Q-NF8） |

### 2.8 実装方針

- `backend-01-architecture.md` R-ARCH-003（バッチの DB 直更新禁止）に従い、バッチは `job/` モジュールの UseCase（例: `PurgeDeletedJobsUseCase`）を経由してドメインルールを通す。バッチ実行クラスから Repository を直接呼び出してドメインルールを迂回しない。
- 時刻取得は `Clock` インターフェース経由に統一する（R-ARCH-010）。

## 3. 通知物理削除バッチ 詳細設計

### 3.1 目的・トリガ

`notifications.created_at <= now() - 90日` を満たす通知（保管期間 90 日、Q-NF10）を対象に物理削除する。

### 3.2 実行スケジュール

- cron 式で深夜 2:00〜4:00 の間に、案件物理削除バッチ（EXT-010）とは別ジョブとして 1 回実行する（対象テーブルが異なるためロック競合なし）。
- 具体時刻はインフラ構築フェーズで確定する（Q-NF9 の枠内であればよい）。

### 3.3 トランザクション単位

- `idx_notifications_created_at`（`tables/notifications.md`）を用いて対象を抽出し、**500 件単位のチャンク**で 1 トランザクション DELETE を行う（通知は案件よりも蓄積量が多くなりうるため、EXT-010 の 1 件 1 トランザクションとは異なりチャンク分割する）。
- 1 チャンクの失敗が他チャンクの処理をブロックしない（部分的な成功を許容する）。

### 3.4 再実行・冪等性

- 冪等: 削除条件（`created_at <= now() - 90日`）を満たさなくなった対象（＝既に削除済み）は再実行時に抽出されないため、同一対象への重複実行は発生しない。
- 失敗したチャンクは翌日以降の実行で再度抽出・再試行される（削除条件を満たし続ける限り）。

### 3.5 分割方針

- 3.3 のとおり 500 件単位のチャンク分割を必須とする（想定データ規模は案件より大きくなりうるため、EXT-010 と異なり第 1 版から分割を組み込む）。

### 3.6 監視連携

- 実行結果（削除件数・実行時間）をアプリケーションログに記録する。
- バッチの継続失敗（複数日にわたる失敗）を検知した場合は運用担当者へメール通知する（Q-NF8、EXT-010 と同じ監視経路。`運用設計.md` 参照）。

### 3.7 失敗時詳細

| 失敗パターン | 挙動 |
|------------|------|
| 特定チャンクの削除処理中の例外 | 当該チャンクのトランザクションのみロールバック。ログにエラー記録。他チャンクの処理は継続する |
| バッチ全体の起動失敗（DB接続不可等） | 当日の実行をスキップし、翌日の実行枠で再試行する |
| 継続的な失敗（複数日） | 死活監視で検知し運用担当者へメール通知（Q-NF8） |

### 3.8 実装方針

- `backend-01-architecture.md` R-ARCH-003（バッチの DB 直更新禁止）に従い、バッチは `notification/` モジュールの UseCase（例: `PurgeExpiredNotificationsUseCase`）を経由してドメインルールを通す。
- 時刻取得は `Clock` インターフェース経由に統一する（R-ARCH-010）。

## 4. バッチなしの明示

案件物理削除バッチ（EXT-010）・通知物理削除バッチ（EXT-011）以外に、第 1 版で必要なバッチは存在しない。以下は第 1 版のスコープ外であり、バッチとして設計しない。

- 完了確認の自動督促・自動確認（Q-J6 決定済み、将来課題）
- 評価未登録の自動遷移（Q-J7 決定済み、期限を設けない）
- マスタデータの定期同期（`移行要件.md` の初期投入は一度きりのデータロードであり、バッチではない）
