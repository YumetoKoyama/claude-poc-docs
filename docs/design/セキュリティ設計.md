# セキュリティ設計

> ID 凡例: [docs/凡例.md](../凡例.md) 参照
> 認証・認可の正典は本ファイル。認可設計（operationId × ロール × テナント条件）は本ファイル 5 節に内包する（`認可設計.md` へは分割しない）。

## 1. 認証方式

- 方式: ログイン ID・パスワードによる認証（要件 `functional/認証.md`）。トークンは JWT（Bearer）を用いる。
- 実装: Spring Security（`backend-00-stack.md` #9）。認証情報の取得は `SecurityContextHolder` / `@AuthenticationPrincipal` 経由に統一し、リクエストヘッダから直接 JWT をパースしない（`backend-07-security-coding.md` R-SEC-031）。

## 2. JWT ライフサイクル（確定値）

| 項目 | 確定値 | 根拠 |
|------|--------|------|
| アクセストークン有効期限 | 30 分 | セッション有効時間（アイドル30分、Q-NF5）に整合させる |
| リフレッシュトークン絶対有効期限 | 8 時間 | 最大セッション長 8 時間（Q-NF5） |
| リフレッシュトークンの保存方式 | STATELESS（アクセストークンは署名検証のみでサーバー側に状態を持たない）。リフレッシュトークンのみ `refresh_tokens` テーブルにハッシュ値で保存し、失効管理を行う | `tables/refresh_tokens.md` |
| 失効方針 | アクセストークン自体は失効機構を持たない（STATELESS のため30分で自然失効）。ログアウト時はリフレッシュトークンを `revoked_at` 設定により失効させる（アクセストークンはクライアント側で破棄） | `auth.yaml` `logout` |
| トークンローテーション | 第 1 版では行わない（同時ログイン制御を行わない方針、Q-NF5、と整合し複数 refresh_tokens 行の同時存在を許容） | `tables/refresh_tokens.md` |
| JWT シークレット管理 | 環境変数 / 外部設定から注入（ソースコードへのハードコード禁止） | `backend-07-security-coding.md` R-SEC-032 |
| JWT ペイロード | `sub`（userId）、`tenantId`、`tenantType`、`iat`、`exp`。パスワードハッシュ等の機微情報は含めない | R-SEC-050 |

## 3. セッション・ロックアウト

| 項目 | 確定値 |
|------|--------|
| ログイン失敗ロック | 連続 5 回失敗で 15 分間ロック（`users.failed_login_count` / `locked_until`、Q-NF2） |
| ロック保存先 | DB（`users` テーブル、Redis 等の別ストアは導入しない。第 1 版は単一インスタンス想定のためテーブル保存で十分、`方式設計.md` 6節） |
| セッション有効時間 | アイドル 30 分でタイムアウト（アクセストークン期限と一致）、最大セッション長 8 時間（Q-NF5） |
| 同時ログイン制御 | 行わない（同一ユーザーの複数端末同時ログインを許可、Q-NF5） |
| ユーザー列挙防止 | ログイン失敗時は ID・パスワードいずれの誤りかを区別しない一律メッセージ（MSG-015）。パスワードリセット申請は登録有無に関わらず常に 202 を返す（`auth.yaml` `requestPasswordReset`） |

## 4. パスワード・暗号化方針

| 項目 | 確定値 |
|------|--------|
| ハッシュアルゴリズム | BCrypt、コスト係数 12 以上（`backend-07-security-coding.md` R-SEC-033、`tables/users.md`） |
| パスワードポリシー | 最小 8 文字以上、英字（大文字小文字問わず）と数字を各 1 文字以上含む（Q-NF4）。有効期間による強制変更は行わない |
| 通信暗号化 | HTTPS 必須（`非機能要件.md`）。証明書運用はインフラ構築フェーズの責務 |
| 時刻取得 | `Clock` インターフェース経由に統一し `LocalDateTime.now()` を直接呼ばない（`backend-01-architecture.md` R-ARCH-010）。ロック解除判定・トークン有効期限判定・応募締切判定等の監査可能性を担保する |

## 5. 認可設計

### 5.1 認可方針サマリ

- 第 1 版はテナント内「同権限のみ」（BR-003）。認可判定はテナント境界（自テナント／他テナント）のみを対象とし、`@PreAuthorize` に集約する（UseCase 内の手書きロールチェックと二重管理しない、`backend-07-security-coding.md` R-SEC-030）。
- **テナント越境（自テナントが当事者でないデータへのアクセス）は 404**（リソース存在の秘匿）、**自テナント内の権限不足（該当しないが第1版では原則発生しない）は 403** に統一する（R-SEC-041）。
- テナントフィルタは SELECT だけでなく UPDATE / DELETE / COUNT / 集計クエリにも適用する（R-SEC-010）。パス変数の ID は信用せず、UseCase 層で「取得 → テナント一致確認」を必ず行う（R-SEC-011）。
- 認可チェックとデータ操作は同一トランザクション内で行い、TOCTOU を防止する（R-SEC-100）。

### 5.2 ロール定義

| ロール | 判定方法 |
|-------|---------|
| `REQUESTER`（配送依頼企業ユーザー） | JWT の `tenantType == REQUESTER` |
| `CARRIER`（運送会社ユーザー） | JWT の `tenantType == CARRIER` |
| 未認証 | JWT なし（public エンドポイントのみアクセス可） |

| `requestPasswordReset` | POST | /api/auth/password-reset-requests | public | — |
| `resetPassword` | POST | /api/auth/password-reset | public（トークンが認可情報） | トークン発行対象ユーザーのみ有効 |
| `registerTenant` | POST | /api/tenants | public | — |

### 5.4 実装方針

- `@PreAuthorize("hasRole('REQUESTER')")` / `@PreAuthorize("hasRole('CARRIER')")` はロール制約がある operationId にのみ付与し、それ以外は `@PreAuthorize("isAuthenticated()")` とする（R-SEC-030）。
- テナント一致確認は Controller ではなく UseCase 層で「取得 → テナント一致確認 → 404 throw」の順で実装する（R-SEC-011、`backend-02-layers.md` の例）。
- 募集中案件の全テナント公開（`listJobs`/`getJobById` の `CARRIER` 参照）は越境ではなく仕様上の公開範囲であるため、テナント一致確認をスキップしてよい（`権限マトリクス.md` 5節）。

## 6. CORS

| 項目 | 確定値 |
|------|--------|
| 許可オリジン | FE のデプロイ先オリジン（本番相当環境確定後に環境変数で設定。ワイルドカード `*` は使用しない、R-SEC-091） |
| 許可メソッド | GET, POST, PATCH, DELETE, OPTIONS |
| 許可ヘッダ | `Authorization`, `Content-Type` |
| Credentials | 不要（JWT は Authorization ヘッダで送信し Cookie を用いないため `Access-Control-Allow-Credentials` は false） |

## 7. CSRF 対策

JWT を Authorization ヘッダで送信し Cookie ベース認証を用いないため、CSRF の主要リスクは低減される。Spring Security の CSRF 保護は REST API（ステートレス）構成のため無効化してよい（`backend-07-security-coding.md` R-SEC-090）。

## 8. 入力サニタイズ・XSS 対策

- BE: Request DTO への Bean Validation（`x-validate-not-blank` 等の YAML ベンダー拡張、R-SEC-020）。自由入力文字列（案件備考・メッセージ本文等）は `pattern: '^[^<>]*$'` 等で HTML タグの混入を制限する（各 `api/*.yaml` 参照）。
- FE: ユーザー入力・API 由来の文字列を HTML として解釈させず、`dangerouslySetInnerHTML` を使用しない（`frontend-coding-rules.md` 5節）。詳細は `フロントエンド共通設計.md` を参照。
- **自由入力フィールドの分類（プレーンテキスト固定）**: 案件備考（`jobs.yaml` `remarks`）・メッセージ本文（`negotiations.yaml` `MessageCreateRequest.body`）・評価コメント（`ratings.yaml` `comment`）は、いずれも「プレーンテキスト（HTML タグ不可）」として扱う。上記の `pattern: '^[^<>]*$'` はこの分類を機械的に強制するものであり、リッチ HTML 入力は第1版のスコープ外とする。

## 9. パスワードリセットのレート制限

- 同一メールアドレスに対し 1 時間あたり 5 回まで（`auth.yaml` `requestPasswordReset`、429 TooManyRequests）。
- メール送信はトランザクション外実行とする（`IF定義.md` EXT-001、送信失敗が業務操作の成功可否に影響しない設計）。

## 10. 機微情報の取り扱い

- パスワード・トークン類は Response DTO に含めない（`writeOnly: true` / `x-sensitive: true`、R-SEC-110/111）。ログにも出力しない（R-SEC-050/051）。
- エラーレスポンスに内部情報（スタックトレース・SQL・テーブル名）を含めない（R-SEC-040、詳細は `共通部品設計.md`）。
