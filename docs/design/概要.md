# 設計概要

> ID 凡例: [docs/凡例.md](../凡例.md) 参照

## 1. スコープ要約

本設計書は、要件定義書（`docs/requirements/`、2026-07-10 時点で `main` にマージ済み・採択済み）に基づき、配送マッチング Web アプリ（配送依頼企業 × 運送会社）の設計成果物一式を定義する。

対象範囲は要件 `概要.md` のスコープをそのまま踏襲する。

- 企業アカウント（テナント）登録・ユーザー追加・ログイン認証・パスワードリセット
- 案件登録・一覧・詳細閲覧・編集・削除（募集中・交渉中のみ）
- 運送会社による応募（単体・セット応募）
- 連絡機能によるメッセージ・金額交渉、最終条件提示・合意による成約処理（スナップショット保存）
- 運送開始報告・完了報告・完了確認によるステータス管理
- 双方の★評価登録
- 通知（画面内アイコン・一覧）
- 取引履歴の閲覧
- 案件の論理削除・物理削除（夜間バッチ、EXT-010）

対象外（設計しない）は要件 `概要.md` の「対象外」節（決済機能、成約後のキャンセル/ペナルティ、ロール分離・承認フロー、二要素認証、リアルタイム Push 通知 等）をそのまま踏襲する。

## 2. 前提条件

- 技術スタックは各子リポジトリの `.claude/rules/*-00-stack.md` を正典とし、本書では再掲しない（フロントエンド: React、バックエンド: Spring Boot、DB: PostgreSQL 16。詳細はスタック確定表を参照）。設計着手前の `check-stack-decided.sh` ゲートは通過済み。
- FE/BE 境界は OpenAPI 3.1（`docs/design/api/`）で定義される REST API とし、認証は JWT（Bearer）ベースとする。
- テナント種別（配送依頼企業 / 運送会社）は登録後変更不可（要件 `アカウント登録.md` 前提条件）。
- 第 1 版はテナント内「同権限のみ」（BR-003）。設計上の認可判定はテナント境界（自テナント／他テナント）のみを対象とし、テナント内のロール分岐は行わない。
- 通知（ENT-009）の宛先粒度は**テナント単位**とする（ユーザー単位ではない）。要件 `権限マトリクス.md` はユーザー単位の「自身宛て」を前提とした記述だが、第 1 版が同権限のみ（BR-003）でありテナント配下の全ユーザーが同一の業務コンテキストを共有するため、設計判断としてテナント単位の通知・既読状態共有を採用する（`tables/notifications.md` 参照）。将来ロール分離導入時はユーザー単位への分離を再検討する。
- 権限マトリクス.md の ENT-001（テナント情報）「R/U（自社テナント情報のみ）」について、更新 UI に対応する画面が要件 `画面一覧.md` に存在しないため、第 1 版は参照（`getMyTenant`）のみを設計する。テナント情報の更新機能は将来拡張とする。
- テナント越境は 404、自テナント内の権限不足は 403 に統一する（`セキュリティ設計.md` 参照）。
- DB マイグレーションはデモ期間の例外として `ddl-auto` 運用とする（`claude-poc-backend/.claude/rules/backend-00-stack.md` #8 を正典とする）。

## 3. ユースケース対応

要件定義の UC-001〜UC-022（`docs/requirements/概要.md` 4節）をすべて設計対象とする。UC ↔ 画面（SCR-XXX）↔ シーケンス（SEQ-XXX）↔ API（operationId）の対応は各画面 md・シーケンス md・`api/*.yaml` の `x-ac` 拡張を参照。俯瞰図は要件 `ユースケース図.md` を参照し、本書では再掲しない。

## 4. リスク

| # | リスク | 影響 | 対応方針 |
|---|-------|------|---------|
| 1 | 応募枠上限（20 社）・応募締切（from の2時間前）・成約処理（セット連動含む）は競合頻度が高い操作であり、read-modify-write では二重許可・上限超過が発生し得る | データ整合性 | 悲観ロック（`SELECT ... FOR UPDATE`）＋ DB 一意制約で保証する（`tables/jobs.md` 排他制御節、`sequences/交渉合意成約.md`・`sequences/セット応募一括合意.md` のロック取得順序参照） |
| 2 | セット応募の連鎖クローズ（BR-006, BR-015）は複数案件・複数応募を同一トランザクションで更新するため、ロック順序を誤るとデッドロックが発生し得る | 可用性 | 対象案件群を ID 昇順で一括ロックする方針に統一（`sequences/セット応募一括合意.md` 参照） |
| 3 | 通知の宛先粒度をテナント単位とした設計判断（本書「前提条件」参照）は、将来ロール分離導入時に見直しが必要になる | 将来拡張コスト | Decision Log（`docs/process/decision-log.md`）へ記録することを人手レビュー時に推奨する |
| 4 | デモ期間の `ddl-auto` 運用は、本番運用移行時に Flyway 等への切り替えが必要 | 運用移行コスト | `DB定義.md` migration 方針に解除条件を明記済み |

## 5. 前提と未解決事項

- 本書作成時点で、要件定義の「設計着手前クローズ必須」の Q-ID（Q-NF*, Q-DM*, Q-A*, Q-J* 等）はすべて `オープン課題.md` で closed 済みであることを確認済み（2026-07-10 時点）。設計を確定させない未解決事項は現時点でない。
- `docs/architecture/constitution.md` は本リポジトリに存在しないため、Constitution Check ゲートはスキップした。
- 通知の宛先粒度（テナント単位）、テナント情報更新 API の未設計（参照のみ）は上記「前提条件」節のとおり設計判断として明示済み。追加の意思決定が必要な場合は `docs/process/decision-log.md` への記録を人手レビュー時に検討する。

## 6. レビュー観点

人手レビュー時に重点確認を推奨する項目:

1. 通知の宛先粒度をテナント単位とした設計判断（4節）が業務要件と整合しているか。
2. テナント情報更新 API を第 1 版で設計しない判断（4節）が妥当か。
3. 応募枠・締切・成約処理の並行制御方針（悲観ロック＋一意制約、`tables/jobs.md`）がパフォーマンス要件（最大同時アクセス100ユーザー）と両立するか。
4. `セキュリティ設計.md` のテナント越境（404）／権限不足（403）の使い分けが全 operationId で一貫しているか。
